    • 🛡️ 应用安全评估实战防守手册 (AppSec Defense Manual)

      本手册基于《应用安全评估技术规范》进行实战化提炼，旨在记录核心漏洞的攻防机理、检测手段及服务端防御闭环。

      📅 第一阶段：身份认证与权限控制

      1.1 认证功能失效 (Broken Authentication)

      1.1.1 登录绕过 (Login Bypass)

      • 攻击机理： 服务端未能统一实行“强校验鉴权”，或者逻辑上过分信任客户端传来的参数。例如在前端或请求中依赖 authenticated=yes 等隐藏域参数，攻击者只需通过抓包改写即可获得合法身份。
      • 检测手段：
        1. 网络嗅探：查找是否存在明文用户名密码传输。
        2. 参数篡改：利用 Burp Suite 拦截登录包，修改类似 user_id、role、status 或 authenticated 等参数进行重放。
        3. 路径扫描：直接访问后台敏感 URL，检查是否会被强制重定向回登录页。
      • 防御闭环 (Gold Standard)：
        • 黄金法则： 绝对不要信任来自客户端的任何关于“是否已认证”的声明。
        • Session 机制： 状态必须存储于服务端会话中。客户端仅持有随机、高熵、不可预测的 SessionID（通过强安全的随机数生成算法）。
        • 全局拦截器： 所有需要鉴权的 URL 必须经过统一的过滤器或拦截器进行会话有效性校验，未登录则强制跳转。

      1.1.4 & 1.1.5 SSO 与 OAuth 认证缺陷（绑定劫持）

      • 攻击机理：
        在 OAuth 快捷登录/绑定流程中，由于缺乏唯一且防重放的会话上下文校验，攻击者可诱导受害者点击预先生成的授权链接。受害者的账户会在不知情的情况下绑定攻击者的社交账户（如微信、Github 等），导致账号失控。
      • 典型劫持场景：
        攻击者点击“绑定微信”，获取到带有微信临时授权码 code 的回调请求后将其拦截并存留。攻击者诱导已登录的受害者访问此回调 URL。由于受害者处于登录状态，系统误认为这是受害者本人发起的回调，最终将受害者的本地账号绑定到攻击者的微信上。
      • 防御闭环 (Gold Standard) —— state 双向绑定与单次作废机制：
        1. 发起请求时（Session 绑定）：
          当用户在浏览器端点击“绑定微信”时，后端必须生成一个高强度的随机安全字符串作为 state（例如：state = crypto.randomUUID()），并将此值存入当前用户的 Session（或加密的单次 Cookie）中。然后将此 state 作为参数，随重定向请求发送给微信：
          https://open.weixin.qq.com/...\&state=UUID\_123
        2. 第三方回调时（一致性校验）：
          微信完成验证后，会将用户重定向回我们的业务后端，并携带 code 和原封不动的 state：
          GET /api/oauth/callback?code=CODE_abc&state=UUID_123
        3. 后端处理（立即作废）：
          • 第一步（对齐校验）： 后端接收到回调后，必须比对 URL 中的 state 是否与当前用户 Session 中存储的 state 严格一致。
          • 第二步（单次销毁）： 无论校验成功还是失败，后端必须立即在 Session 中销毁（作废）该 state。
          • 第三步（执行绑定）： 校验通过后，后端再拿着 code 去微信换取 access_token 完成绑定。

      1.2 认证信息猜解 (Credential Guessing)

      1.2.1 & 1.2.2 暴力破解与弱口令 (Brute Force & Weak Credentials)

      • 攻击机理：
        应用在登录认证入口未限制尝试频率、未实施验证码或账号锁机制，导致攻击者可以通过自动化脚本对账号密码进行大批量碰撞猜解。
      • 检测手段：
        抓取登录请求包发送至 Burp Suite Intruder，使用预置密码字典（如 Top100 弱口令、admin 等）对目标账号进行穷举。
      • 防御闭环 (Gold Standard)：
        • 自适应认证机制： 1. 验证码一次性销毁： 验证码不仅要随机且增加噪声，最关键的是服务端在接收到校验请求后（不论成功与失败）必须立即将该验证码从 Session 中销毁，彻底杜绝验证码回放。
          2. 账号锁定策略： 同一账号在 5 分钟内连续失败 6 次，则强制锁定该账号 3 小时，或强制弹出滑块验证码。
          3. 多因素认证 (MFA)： 针对异地登录、高敏业务或高风险环境，强制启用短信/邮箱二次验证。

      1.4 认证功能滥用 (Authentication Misuse)

      1.4.2 多点认证缺陷（多设备登录限制）

      • 攻击机理：
        系统允许同一账号在多个浏览器、设备或 IP 同时登录。如果账号密码泄露，攻击者登录时，合法用户不会收到任何中断提示或挤退通知，导致凭证盗用处于无感知状态。
      • 检测手段：
        在浏览器 A 中登录账号 X，同时在浏览器 B 中使用相同账密登录，观察浏览器 A 的会话是否依然有效。若均可正常操作，则存在多点认证缺陷。
      • 防御闭环 (Gold Standard) —— 单一会话挤退控制：
        在关键业务系统（如涉敏系统）中，在服务端内存或 Redis 中以用户 ID（uid）为 Key 维护当前激活的 SessionID：
        1. 用户在设备 B 登录时，服务端生成全新的 SessionID_B。
        2. 写入存储时，覆盖原先的 SessionID_A，并将旧的 SessionID_A 标志为作废。
        3. 当浏览器 A 下一次发送请求时，拦截器识别到其持有的是已被作废的 Session，强制销毁本地凭证并清空状态，同时弹出提示：“您的账号已于其他设备登录，请注意账户安全”。

      1.5 防护功能缺失 (Defensive Controls Lack)

      1.5.5 Session 会话注销缺陷

      • 攻击机理：
        用户点击“注销/退出”时，后端仅返回重定向状态而没有在服务端对会话数据进行真实的内存销毁。在无状态 Token（如 JWT）架构中，前端执行 removeItem 只是掩耳盗铃，该 Token 在有效期内依然被全局承认。
      • 检测手段：
        在正常操作界面点击“注销/退出”，拦截退出后的重定向包，再次向系统后台接口（如 /api/user/info）发送该注销前的 SessionID/Token，观察是否依然可以获取到敏感数据。
      • 防御闭护 (Gold Standard)：
        • 有状态 Session : 服务端必须调用 session.invalidate() 显式将内存中的会话数据擦除，并将关联的 Cookie 清空。
        • 无状态 Token (JWT) : 引入后端黑名单机制（使用 Redis 存储，失效时间为 Token 剩余有效期）。拦截器对每个接收到的 Token 在黑名单内比对，若命中则立即拒绝访问。

      1.5.7 Session 会话固定 (Session Fixation)

      • 攻击机理：
        服务器在用户未登录到已登录的状态转换（Authentication State Transition）期间，复用了同一个会话标识符（Session ID），从而允许攻击者预先生成匿名会话标识并强加给受害者，实现劫持登录。
      • 检测手段：
        1. 记录未登录状态下浏览器的 SessionID。
        2. 完成登录，再次查看当前的 SessionID，比对是否相同。
        3. 若相同，则说明存在会话固定漏洞。
      • 防御闭环 (Gold Standard) —— 强制重建会话机制 (Regeneration)：
        • 服务端验证逻辑中，一旦用户名和密码比对通过，首先调用当前会话的销毁方法（如 Servlet 中的 request.getSession().invalidate()）。
        • 随后立即调用 request.getSession(true) 强制在服务端生成一个全新的、与之前的匿名状态完全无关联的 Session ID 颁发给客户端。

      1.6 防护功能失效 (Defensive Controls Failure)

      1.6.7 Response 状态值修改 (Response Manipulation / Client-Side Trust)

      • 攻击机理：
        前后端架构或移动端 APP 设计中，前端仅仅根据后端返回的响应体状态字段（如 {“result”: “success”, “code”: 1}）来控制前端页面的跳转和管理菜单的显示，而没有在后续数据交互时将当前认证凭证（如 Session Cookie/Token）发送给服务端进行强制鉴权。攻击者通过拦截响应包并修改状态字段，可直接骗过前端 UI 绕过验证机制（如验证码或登录限制）。
      • 检测手段：
        1. 输入错误的账号、密码或故意输错验证码，提交并用代理工具拦截响应包。
        2. 将响应包体中的错误信息/失败状态（如 fail、code: 0）修改为对应的成功状态（如 success、code: 1）并释放响应。
        3. 观察前端是否绕过了限制直接跳转至登录后或下一步操作页面。
      • 防御闭环 (Gold Standard)：
        • 决策权上卷服务端： 前端不决定状态，只展示状态。
        • 无状态/有状态鉴权强校验： 所有受保护的后台菜单、功能接口和高敏操作 API（增、删、改、查），必须独立校验请求头/Cookie 中的会话凭证。即便客户端界面被强行跳转，若 API 请求时未携带在服务端判定为“已认证”的合法 SessionID/Token，服务端必须一律拦截并返回 401 Unauthorized 或 403 Forbidden。

      1.6.1 & 1.6.3 验证码参数置空与弱类型比对漏洞 (Verification Parameter Bypass)

      • 攻击机理：
        服务端在对验证码字段（如 sms_code）进行一致性对比时，未对参数的“存在性”与“类型”进行严格的安全前置过滤，导致因为弱类型对比隐含的“逻辑恒真”或代码异常流默认通过，使攻击者可以通过删除该参数直接绕过验证。
      • 检测手段：
        将验证码字段（如 sms_code）的值置空、赋予不匹配类型（如数组 []）、或直接将该参数在 JSON 请求体中删除，观察是否能绕过验证直接进入下一个业务环节。
      • 防御闭环 (Gold Standard) —— “前置白名单强校验”与“Fail-Secure原则”：
        1. 严格的入参非空校验： 在进入比对逻辑前，首先强校验参数是否存在，不仅不能为 null，也不能为空白字符。
        2. 严格的强类型对比： 必须使用强等于（如 PHP/JS 的 ===，Java 中的 sessionCode != null && sessionCode.equals(inputCode)）。
        3. Fail-Secure (失效安全原则)： 初始化状态必须为 authorized = false。任何 catch 分支或异常流程必须默认设为“拒绝访问”。

      📅 第二阶段：安全防护与业务逻辑

      1.8 业务逻辑篡改 (Business Logic Flaws)

      1.8.5 竞争条件 (Race Condition / 条件竞争)

      • 攻击机理：
        在高并发的多线程环境下，程序执行“检查凭证（Time of Check）”与“执行变更（Time of Use）”之间存在时间差（TOC/TOU 冲突）。若服务端缺乏并发锁机制，攻击者利用自动化脚本在微秒级时间内并发发送大量请求，可使多个线程同时查到“未限制”状态，从而突破“限领一次”或“库存扣减”等业务限制。
      • 检测手段：
        使用 Burp Suite Intruder 的 Null payloads 模式，并发线程设置为 20 以上，或者编写 Python 多线程脚本同时向目标接口（如领券、抽奖、提现）发送并发请求，观察是否实现多券领取或超额变现。
      • 防御闭环 (Gold Standard)：
        • 悲观锁控制：在数据库查询语句中对操作行加锁（如 SQL 使用 FOR UPDATE），强迫高并发请求在数据库层面“串行排队”。
        • 消息队列削峰串行化：在最前端使用 Redis 队列（FIFO 机制）承接请求，由单线程消费者逐个处理，将网状高并发彻底拉直。

      1.8.8 正负值对冲与数据篡改漏洞 (Negative Value Manipulation)

      • 攻击机理：
        系统过分信任来自前端客户端提交的敏感价格、数量、或折扣参数，在后端计算最终账单时，未对入参的数量范围（如是否大于0）进行边界检查。攻击者通过传入负数数量（如购买商品数量为 -2）使系统执行乘以负数的数学运算，从而冲平其他商品的总价，实现“零元购”或“逆向充值”。
      • 检测手段：
        在购物车结算或调用支付接口时抓包，修改商品数量（quantity）为负数（-1，-500）、或者修改单价（price）为极低值、修改抵扣积分（points）为负数，观察账单最终应付总额是否发生对冲减少。
      • 防御闭环 (Gold Standard) —— “数据流向可信链原则”：
        1. 输入合法性边界判定：后端接收到任何带有数量（quantity）的入参时，必须实行强硬的数学边界检查（必须满足 quantity > 0 且为正整数），任何非正整数一律抛出异常并阻断业务。
        2. 绝对剥离前端价格决定权：前端提交的购物车数据包严禁携带价格（price）字段。后端必须仅接收商品 ID（item_id），由后端服务在结算时亲自向可信的数据库查询真实单价，并在服务器内存中完成账单总价的最终算术计算。
