🛡️ 应用安全评估实战防守手册 (AppSec Defense Manual)

本手册基于《应用安全评估技术规范》进行实战化提炼，旨在记录核心漏洞的攻防机理、检测手段及服务端防御闭环。

📅 第一阶段：身份认证与权限控制

1.1 认证功能失效 (Broken Authentication)

1.1.1 登录绕过 (Login Bypass)

  • 攻攻机理： 服务端未能统一实行“强校验鉴权”，或者逻辑上过分信任客户端传来的参数。例如在前端或请求中依赖 authenticated=yes 等隐藏域参数，攻击者只需通过抓包改写即可获得合法身份。
  • 检测手段：
    1. 网络嗅探：查找是否存在明文用户名密码传输。
    2. 参数篡改：利用 Burp Suite 拦截登录包，修改类似 user_id、role、status 或 authenticated 等参数进行重放。
    3. 路径扫描：直接访问后台敏感 URL，检查是否会被强制重定向回登录页。
  • 防御闭环 (Gold Standard)：
    • 黄金法则： 绝对不要信任来自客户端的任何关于“是否已认证”的声明。
    • Session 机制： 状态必须存储于服务端会话中。客户端仅持有随机、高熵、不可预测的 SessionID（通过强安全的随机数生成算法）。
    • 全局拦截器： 所有需要鉴权的 URL 必须经过统一的过滤器或拦截器进行会话有效性校验，未登录则强制跳转。

1.2 认证信息猜解 (Credential Guessing)

1.2.1 & 1.2.2 暴力破解与弱口令 (Brute Force & Weak Credentials)

  • 攻攻机理：
    应用在登录认证入口未限制尝试频率、未实施验证码或账号锁机制，导致攻击者可以通过自动化脚本对账号密码进行大批量碰撞猜解。
  • 检测手段：
    抓取登录请求包发送至 Burp Suite Intruder，使用预置密码字典（如 Top100 弱口令、admin 等）对目标账号进行穷举。
  • 防御闭环 (Gold Standard)：
    • 自适应认证机制： 1. 验证码一次性销毁： 验证码不仅要随机且增加噪声，最关键的是服务端在接收到校验请求后（不论成功与失败）必须立即将该验证码从 Session 中销毁，彻底杜绝验证码回放。
      2. 账号锁定策略： 同一账号在 5 分钟内连续失败 6 次，则强制锁定该账号 3 小时，或强制弹出滑块验证码。
      3. 多因素认证 (MFA)： 针对异地登录、高敏业务或高风险环境，强制启用短信/邮箱二次验证。

1.9 权限篡改 (Privilege Escalation)

1.9.2 水平越权 (Horizontal Privilege Escalation)

  • 攻攻机理：
    相同角色/权限等级的不同用户之间，系统未能建立“数据所有权与当前会话身份”的强绑定。直接信任了客户端传入的 uid 或 order_id 等直接引用。
  • 典型场景：
    访问 POST /api/user/updateAvatar?uid=8888，攻击者直接篡改参数 uid=8889 即可越权修改他人头像。
  • 检测手段：
    注册两个同等权限的测试账号 A、B。在账号 A 登录状态下抓包，将请求参数中的身份标识（如 ID）修改为账号 B 的标识，重放后观察是否能成功修改或查看账号 B 的敏感信息。
  • 防御闭环 (Gold Standard)：
    • 数据与身份强绑定 (ID-to-Session Binding)： 服务端在进行数据查询、修改或删除操作时，严禁从客户端请求参数中获取用户 ID，而应当直接从当前已经通过安全认证的 Session/Token (如解密后的 JWT) 中提取当前登录用户的唯一标识，并在此基础上执行数据库的 SQL 语句（例如：WHERE user_id = :session_user_id）。

1.9.1 垂直越权 (Vertical Privilege Escalation)

  • 攻攻机理：
    低权限用户（如普通员工）成功访问并执行了本属于高权限用户（如系统管理员）的受控页面或接口。多见于后台菜单仅做前端隐藏，但后端未进行针对具体 API 接口的角色授权控制（RBAC）。
  • 典型场景：
    普通用户 Alice 抓包发现 Cookie 带有可读的属性 role=normal。当她篡改其为 role=admin 并刷新页面时，直接进入了后台管理界面。
  • 防御闭环 (Gold Standard)：
    • 会话变量加密与签名 (JWT 规范)：
      如果必须在客户端 Token 中存储角色（role）或身份状态，必须采用行业标准的 JSON Web Token (JWT) 方案，并启用强加密算法（如 RS256）。后端每次接收请求时，必须校验 Token 的签名完整性，任何客户端试图将 normal 改为 admin 的行为，都会由于没有服务端的私钥导致签名不匹配，从而被立刻识破和拦截。
    • 后端鉴权控制（基于角色的访问控制 - RBAC）：
      对每一个后端 API 进行硬性的基于权限的注解控制（例如 Spring Security 中的 @PreAuthorize(“hasRole(‘ADMIN’)”)）。
