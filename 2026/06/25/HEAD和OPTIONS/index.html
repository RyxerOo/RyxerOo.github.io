一、HEAD

1、概念

HEAD请求是HTTP协议中的一种请求方法，与GET请求类似，但不返回实际的响应主体。它主要用于获取资源的元数据，检查资源是否存在以及验证资源是否被修改。

deepseek说：“HEAD是一种与 GET 方法完全相同的 HTTP请求方法，但核心区别在于服务器不会在响应中返回消息体（Body），只返回状态码和响应头（Headers）。

简单理解：HEAD 相当于“精简版”的 GET。服务器收到 HEAD 请求后，会像处理 GET 一样执行权限校验、读取数据库等逻辑，但在最终打包发送数据时，只发送“包裹单”（头部信息），而省略了“包裹内容”（实体数据）。”

2、HEAD请求的语法与用法

HEAD请求的语法与GET请求相同，只是请求行中的方法变为HEAD。

例：

HEAD /path/to/resource HTTP/1.1
Host: example.com

3、HEAD请求的响应格式

HEAD请求的响应格式与GET请求相同，只是响应主体为空。响应中包含了与GET请求相同的响应头信息，如Content-Type、Content-Length等。

4、HEAD 的主要用途

因为不需要传输巨大的数据体，HEAD 非常高效，常用于以下场景：

  • 检查资源是否存在：通过返回的 200 或 404 状态码判断文件是否有效，如果服务器返回404 Not Found状态码，则表示资源不存在；如果返回200 OK状态码，则表示资源存在。

  • 获取资源的元数据：通过发送HEAD请求，可以获取资源的响应头中的 Content-Length（文件大小）、Content-Type（文件类型）或 Last-Modified（最后修改时间）等。这对于某些场景下只需要了解资源的基本信息而不需要下载实际内容的情况非常有用。

  • 测试网络连通性：检测服务器是否可达且响应正常。

  • 验证缓存是否过期：配合 If-Modified-Since 等请求头，查看服务器上的资源是否比本地缓存更新。

  • 验证资源是否被修改：通过发送HEAD请求，可以获取资源的ETag或Last-Modified头信息，用于验证资源是否被修改。如果客户端拥有先前获取的ETag或Last-Modified值，并发送到服务器进行比较，服务器可以返回304 Not Modified状态码，表示资源未被修改。

5、HEAD 与 GET 的关键区别

  • 响应体（Body）：GET 返回完整数据；HEAD 绝对不返回 Body，因此数据量比GET请求要小。如果服务器在 HEAD 响应中返回了 Body，必须被客户端忽略或视为错误。
  • 性能：HEAD 节省带宽和传输时间，尤其在处理大文件（如视频、安装包）时优势明显。
  • 幂等性：两者都是安全且幂等的（即多次请求不会改变服务器状态）。
  • 缓存机制的差异：由于HEAD请求不返回实际的响应主体，因此无法被缓存。而GET请求返回的响应可以被缓存，提高性能和减轻服务器负载。

6、注意事项

  • HEAD请求的安全性考虑：在使用HEAD请求时，需要注意安全性考虑。由于HEAD请求可以获取资源的元数据，可能会暴露一些敏感信息，因此需要确保只有授权的用户可以发送HEAD请求。
  • 如何处理HEAD请求的响应：在接收到HEAD请求的响应后，可以根据响应的状态码和头信息进行相应的处理。根据实际需求，可以解析响应头信息、验证资源是否被修改等。

二、OPTIONS

1、概念

HTTP OPTIONS 方法向给定 URL 或服务器请求其允许的通信选项。它可用于测试一个请求允许使用的 HTTP 方法，或用于确定一个请求在发起 CORS 预检请求时是否会成功。客户端可以使用该方法指定一个 URL，或者使用星号（*）来指代整个服务器。

deepseek说：“OPTIONS 是一种 HTTP 请求方法，其核心作用是“探测”服务器对某个特定资源所支持的通信能力（主要是允许的 HTTP 方法），以及跨域资源共享（CORS）策略。

简单理解：OPTIONS 相当于“事前侦察兵”。在正式发起复杂请求之前，客户端（通常是浏览器）先发送一个 OPTIONS 请求，询问服务器：“我打算这么干，你允许吗？”如果服务器回复“允许”，浏览器才会继续发送实际的请求。”

2、主要用途

HTTP OPTIONS 是一种特殊的请求方法，其主要作用是让客户端了解服务器支持的功能。 通过发送一个 OPTIONS 请求到目标资源，客户端可以获取服务器允许的 HTTP 方法列表以及其他相关信息。 例如，当浏览器发起跨域请求时，会先发送一个 OPTIONS 请求（预检请求），以确认实际请求是否符合服务器的安全策略。

  • CORS 预检请求（Preflight）：这是当前最常见的使用场景。当浏览器发起跨域请求，且该请求不是“简单请求”（例如使用了 PUT/DELETE 方法，或携带了自定义请求头如 Authorization），浏览器会自动触发一次 OPTIONS 请求。服务器必须通过特定的响应头（如 Access-Control-Allow-Origin）来授权，否则实际请求会被浏览器拦截。
  • API 接口自描述/发现：在 RESTful 架构中，客户端可以主动向服务端发送 OPTIONS 请求，通过返回的 Allow 响应头，了解该端点支持哪些操作（例如支持 GET、POST、PUT 还是 PATCH）。
  • 测试服务器功能： 开发者可以通过发送 OPTIONS 请求来检查某个接口支持哪些 HTTP 方法和配置，从而验证 API 的正确性。

3、特性

特性维度 核心表现 技术细节与注意事项
安全性与幂等性 只探测，不修改 （安全 & 幂等） • 被设计为只读操作，绝不能改变服务器上的任何资源状态。 • 后端在处理 OPTIONS 路由时，必须禁止执行业务上的增、删、改逻辑。
双模式触发机制 自动预检 和 主动探测 • 自动模式：浏览器针对跨域非简单请求（如 PUT/DELETE）自动触发，开发者无感知。 • 手动模式：开发者可主动编码调用（如 fetch(url, {method:'OPTIONS'})），用于 API 自发现。
响应体（Body） 极少携带实体数据 • 标准响应通常不含消息体（Body 为空）。 • 实际场景中，服务器收到后多直接返回 204 No Content 或 200 OK，旨在极速响应、节省带宽。
结果可缓存性 预检结果可缓存 （性能加速器） • 服务器通过 Access-Control-Max-Age（单位：秒）头部告知浏览器缓存时长（如 7200）。 • 缓存生效期内，跳过后续 OPTIONS 请求，直接发起实际请求，大幅降低网络延迟（RTT）。
执行优先级 拥有一票否决权 • 在跨域场景中，若 OPTIONS 预检失败（未返回正确的 CORS 头或返回 4xx/5xx），浏览器会直接拦截。 • 此时后续的真实业务请求（如 POST/PUT）根本不会被发送，这是它区别于其他 4xx 错误的重要特性。

4、OPTIONS 的响应关键信息

与 HEAD 返回“资源属性”不同，OPTIONS 返回的是“交互规则”，主要看这几个响应头：

  • **Allow**：列出该资源支持的所有 HTTP 方法（例如 Allow: GET, POST, HEAD）。
  • CORS 相关头（预检关键）：
    • Access-Control-Allow-Origin：允许哪些域名访问。
    • Access-Control-Allow-Methods：允许具体哪些方法。
    • Access-Control-Allow-Headers：允许携带哪些自定义头。
    • Access-Control-Max-Age：本次预检结果的有效缓存时间（避免每次请求都发 OPTIONS，提升性能）。
