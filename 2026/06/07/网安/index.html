1. SQL报错注入有哪些函数？

不同数据库的常用报错注入函数：

  • MySQL：updatexml()、extractvalue()、floor() + rand() + group by 主键冲突、exp()（5.5.5之前溢出报错）、name_const() 等。
  • MSSQL：convert(int, @@version) 类型转换报错、cast、在 having / group by 中利用列名暴露表名。
  • Oracle：utl_inaddr.get_host_name()、ctxsys.drithsx.sn()、dbms_xdb_version.makeversioned() 等。
  • PostgreSQL：没有像 MySQL 那样的专用报错函数，但可通过类型转换强制报错，如 ''::int、1/0 等。

2. 查看 80 端口被哪个进程占用，Linux 下用什么命令？

常用三个命令：

  • ss -tlnp | grep :80 # 推荐，更快

  • netstat -tlnp | grep :80 # 传统方法

  • lsof -i :80 # 直接列出进程名和 PID

例如 ss -tlnp | grep :80 会显示类似 LISTEN 0 128 *:80 *:* users:(("nginx",pid=1234,fd=6)) 的输出，直接看到进程名和 PID。

3. 杀死进程用什么命令？具体点。

  • 找到 PID 后使用 kill：

    kill 1234 # 发送 SIGTERM，优雅退出
    kill -9 1234 # 发送 SIGKILL，强制杀死

  • 按名字杀进程：

    killall -9 nginx # 杀死所有 nginx 进程
    pkill -9 -f nginx # 支持正则匹配，-f 匹配完整命令行

6. 中了勒索病毒，从技术层面怎么应急？

不是流程和方法，纯技术操作要点：

  • 立即断网但保持开机：防止横向传播和远程控制，同时避免丢失内存中的密钥或进程信息。
  • 保存现场：
    • 导出内存 dump（用 DumpIt、Magnet RAM Capture），可能提取到密钥。
    • 抓取正在运行的进程列表、网络连接（netstat -ano）、计划任务、服务、启动项、注册表 Run 键等。
    • 保留勒索信、加密文件样本、加密后缀，上传到 ID Ransomware 或 No More Ransom 识别家族。
  • 分析入侵途径：检查系统日志（事件 ID 4625/4624 登录、SMB/3389 日志）、Web 日志、新创建的用户、高危漏洞利用痕迹（如 MS17010）。
  • 寻找解密可能：对于已知家族（如 STOP/Djvu 的部分变种、Maze 泄漏的密钥）可找到解密工具；否则依靠备份或卷影副本。
  • 清除与恢复：确认病毒全部清除后，从干净备份恢复，或重装系统。

7. 内网环境下给你一个 IP，如何进行渗透测试？

技术步骤：

  1. 信息收集：
    • 端口扫描：nmap -sS -sV -p- -T4 识别服务和版本。
    • 漏洞扫描：nmap --script vuln 或 Nessus 扫常见漏洞。
    • 服务弱口令爆破：SMB、RDP、SSH、FTP、数据库等（Hydra、Medusa）。
    • Web 探测：有 Web 服务则扫目录、CMS 指纹，检查常见框架漏洞。
  2. 漏洞利用打点：
    • 高危漏洞优先：永恒之蓝（MS17-010）、BlueKeep、SMBGhost 等。
    • Web 漏洞：SQL 注入、反序列化、Shiro、Fastjson、上传 getshell。
    • 弱口令登录，进一步收集信息。
  3. 后渗透与横向移动：
    • 提权：系统补丁分析、内核漏洞、土豆家族、UAC 绕过。
    • 获取凭据：导出 hash（mimikatz）、浏览器密码、RDP 保存的连接。
    • 横向移动：psexec、wmiexec、smbexec、黄金票据/白银票据、Pass-the-Hash。
  4. 持续控制：创建隧道代理、注册计划任务，逐渐扩大战果。

8 . SQL Server 数据库，对方要在什么配置下才能阻止继续上传木马？

攻击者通常利用 SQL Server 写文件或执行命令来上传木马，可从以下配置阻止：

  • 禁用危险存储过程：

    EXEC sp_configure ‘show advanced options’, 1; RECONFIGURE;
    EXEC sp_configure ‘xp_cmdshell’, 0; RECONFIGURE;

    同样禁用 sp_OACreate、sp_OAMethod 等 OLE 自动化。

  • 最小化权限：

    • 应用连接数据库的账号不要用 sa 或高权限角色；
    • 只赋予必要的 db_datareader、db_datawriter 等；
    • 收回 BULK INSERT、BACKUP DATABASE/LOG 到任意路径的权限。

  • 文件系统权限：

    • SQL Server 服务账户（如 MSSQL$INSTANCE）仅给予数据目录读写权限，禁止对 Web 目录或其他目录写入。
    • 使用软件限制策略或 AppLocker 阻止 EXE 在非预期目录运行。

  • 配置审计与告警：

    • 开启 SQL Server Audit 监控存储过程的调用；
    • 监控服务器上新增文件（如通过 Windows 审计策略或文件完整性监控）。

9. 文件上传 JSP 成功了，但访问文件不执行，为什么？

常见技术原因：

  • 中间件未配置解析 JSP：比如 Nginx 反向代理至 Tomcat，但没有转发 .jsp 请求给 Java 容器。
  • 上传目录没有执行权限：Servlet 容器（Tomcat/WebLogic）中，只有特定应用目录（如 /ROOT 或虚拟目录）内的 JSP 才会编译执行，若上传到静态资源目录或自定义映射不包含 JSP 的位置，则不会执行。
  • Web 应用配置：web.xml 中 <jsp-file> 配置可能禁用了某些路径；或使用了 Spring Boot 嵌入式容器，直接访问 JSP 需满足特定条件（JSP 放在 src/main/webapp/ 且引入 jasper 依赖）。
  • 文件名被改写/权限不够：上传时后缀被改为 .jsp;.jpg 或大小写问题，或者文件所有者不是应用运行用户，导致无法读取。
  • WAF 拦截：请求时被 WAF 过滤，返回 403/404，但文件实际存在。

10. 逻辑漏洞你都了解哪些？详细说明。

  • 越权漏洞：
    • 水平越权：同权限用户通过修改 ID 等参数查看/操作他人数据。
    • 垂直越权：低权限用户执行高权限操作（如普通用户访问管理接口）。
  • 支付逻辑篡改：修改商品价格、数量、优惠券参数为负数或极小值，改支付状态、订单金额。
  • 验证码相关：验证码可重用、验证码在客户端校验、返回在响应包中、绕过机制（不刷新可直接使用）。
  • 密码重置漏洞：验证码爆破、修改接收手机号/邮箱、密码找回答案可猜测、任意用户密码重置（修改 uid）。
  • 条件竞争：并发请求导致的多次操作（如重复领券、重复提现、同一优惠多次使用）。
  • 接口未授权访问：未做身份校验的管理后台、API 接口直接访问。
  • URL 跳转/SSRF：未校验跳转地址，可诱导用户跳转到钓鱼页；SSRF 可打内网。
  • 加密与签名缺陷：弱签名算法、可逆加密、枚举暴力破解 token。
  • 数据校验不全：邮件/手机格式绕过、负数库存、超大数值溢出。

11. 验证码漏洞并发刷票，用什么插件？

推荐 Burp Suite 的 Turbo Intruder 插件（BApp Store 安装）。

  • 单窗口就可以轻松实现 50~100 并发线程，通过 HTTP 管道或异步 IO 发送大量请求。
  • 配合 %s 占位符和字典生成器，可以快速并发爆破或刷票。
  • 也可使用 Python 的 aiohttp + asyncio 编写自定义并发脚本，但 Turbo Intruder 更便捷。

12. CSRF 漏洞原理是什么？

  • 用户登录受信任网站 A，生成 Cookie 等会话凭证。
  • 攻击者诱导用户访问恶意网站 B，B 向 A 发送一个伪造请求（如修改密码、转账），浏览器自动携带用户对 A 的 Cookie。
  • 服务端只验证了 Cookie，没有验证请求来源（Referer/Origin）或没有使用不可预测的 CSRF Token，导致请求被正确处理，以用户身份执行了非预期操作。

13. 信息收集后怎么下一步分配攻击面？

从信息收集中提取出的资产进行分层与优先级排序：

  • 暴露面评估：
    • 对外 Web 应用：是否有已知 CMS 漏洞、通用框架漏洞（Struts2、Fastjson、Shiro、Log4j）；
    • 高危端口服务：SMB、Redis 未授权、Docker API、RDP 弱口令等。
  • 攻击链规划：
    • 最容易突破的点优先：弱口令、命令执行、未授权访问。
    • 次级：注入、XSS 获取凭证、SSRF 打内网。
  • 分区域渗透：
    • 获得初始权限后，梳理内网 IP 段、域控位置、数据库地址，划分为多个渗透区；
    • 利用代理进入内网，执行扫描与横向，逐步扩大权限直至目标。

14. DDoS 攻击打进来怎么做应急？

技术层面：

  • 流量清洗/路由：联系上游 ISP 或云服务商进行黑洞路由或流量清洗；使用高防 IP 或 CDN 散射。
  • 本地缓解：
    • 使用 iptables/nftables 限制每秒连接数、并发连接，DROP 非法包（如 SYN flood 可开启 SYN Cookie）。
    • 调整内核参数：net.ipv4.tcp_syncookies=1、net.core.somaxconn。
    • Nginx/LVS 对 HTTP Flood 限制请求速率（limit_req_zone）。
  • 攻击分析：抓包分析类型（SYN/ACK/HTTP/DNS），找到攻击特征（固定源端口、URL 路径），在 IDS/IPS 层临时阻断。
  • 留存证据：保存 pcap、日志，用于溯源和报案。

15. Java Web 的 Spring Boot 一般有什么漏洞？

  • Actuator 未授权访问：/env、/heapdump、/mappings 暴露配置和敏感信息，/env 可能结合 Eureka 做命令执行（如 spring-cloud-starter-netflix-eureka-client 的 xstream 反序列化）。
  • Spring Cloud Gateway RCE：CVE-2022-22947，通过动态路由添加执行 SpEL 表达式实现 RCE。
  • Spring4Shell（CVE-2022-22965）：在 JDK9+ 下，通过参数绑定写入 Tomcat 日志配置，写入 Webshell。
  • WhiteLabel Error Page 信息泄露：暴露接口和异常信息。
  • Spring Data Rest 相关 RCE：老版本有反序列化或 SpEL 注入。
  • 依赖组件漏洞：Fastjson、Jackson 反序列化、Log4j JNDI 注入等。

16. PHP 编写的 phpMyAdmin 有什么漏洞？

  • RCE：
    • CVE-2016-5734：4.6.x 之前，通过 preg_replace 的 /e 修饰符在 TableSearch 控制器实现代码执行。
    • CVE-2018-12613：4.8.x 文件包含漏洞，通过 target 参数造成 LFI 并可配合写文件 Getshell。
  • 反序列化漏洞：/setup/ 下的反序列化可导致代码执行（老版本）。
  • 弱口令/配置缺陷：默认无密码、空口令 root，暴力破解。
  • 信息泄露：phpinfo() 暴露、/setup/ 未删除、配置文件权限不当。

17. Python 的 Django 有什么漏洞？

  • SQL 注入：当使用 .raw()、.extra() 或 QuerySet.annotate 聚合函数拼接不安全的用户输入时，可能导致注入（如 CVE-2022-34265）。
  • 调试模式信息泄露：DEBUG=True 时，错误页面展示堆栈、环境变量，甚至可暴露出 SECRET_KEY。
  • Session 反序列化/签名缺陷：如果 SECRET_KEY 泄露，攻击者可伪造 Session，配合 Cookie-based 后端（如 django.contrib.sessions.backends.signed_cookies）可能导致任意代码执行（需利用 pickle 序列化，Django 默认用 JSON 较安全，但若使用自定义或历史版本可能风险）。
  • Admin 后台弱口令：Django Admin 暴露出所有模型，弱口令直接控制全部数据。
  • 路径遍历/任意文件读取：在开发中错误地拼接文件路径，如使用 ../ 绕过。
  • CSRF 防护缺失：如果未使用 {% csrf_token %} 或使用了 @csrf_exempt，可能引入 CSRF。
  • Django REST Framework (DRF) 配置不当：过度暴露接口、不安全的 ViewSet 权限（允许未认证操作）、CORS 配置错误等。

18、sqlmap –os-shell 流程

第一阶段：前置检测 判断数据库类型（MySQL、MSSQL、PostgreSQL等） 判断当前用户权限（是否为DBA/超级管理员） 判断注入类型（布尔、报错、堆叠、联合查询等）

第二阶段：选择利用方式（分数据库） 尝试 UDF 提权 判断数据库版本和操作系统（决定UDF库的位数和格式） 判断插件目录位置（通过 select @@plugin_dir 等） 判断是否支持写入文件（secure_file_priv 是否允许） 将UDF库以十六进制形式写入目标服务器 创建自定义函数（CREATE FUNCTION sys_eval RETURNS STRING SONAME ‘udf.dll’） 测试函数是否可用（select sys_eval(‘whoami’)） 如果UDF失败，尝试 WebShell 探测网站绝对路径（常见路径字典爆破） 写入WebShell文件（select ‘‘ into outfile ‘路径’） 通过HTTP请求访问WebShell执行命令 如果是 MSSQL： 检查 xp_cmdshell 状态 如果禁用但有 sa 权限，执行启用命令： EXEC sp_configure ‘show advanced options’,1; RECONFIGURE; EXEC sp_configure ‘xp_cmdshell’,1; RECONFIGURE; 直接调用 xp_cmdshell 执行命令 如果是 Oracle： 创建Java存储过程 通过Java调用系统命令 获取执行结果

第三阶段：建立交互式shell 命令执行测试（执行简单命令如 whoami） 建立交互通道 如果是UDF方式：循环发送SQL请求执行命令，返回结果 如果是WebShell：循环发送HTTP请求，携带命令参数 如果是xp_cmdshell：循环执行并读取结果 处理输出（编码转换、错误处理、分页等）

28、域前置有了解过吗？

  • 定义：一种利用 CDN 或共享基础设施，使 TLS SNI 与 HTTP Host 头指向不同域名，从而隐藏真实通信目标、绕过网络审查的技术。
  • 原理：
    • TLS 握手阶段，SNI 填写一个高信誉、允许访问的域名（如 cdn-trust.com），完成证书验证和加密通道建立。
    • 在加密的 HTTP 请求中，Host 头 填写真正要访问的域名（如 evil-c2.com）。
    • CDN 根据解密后的 Host 头将流量转发到后端，审查者只能看到与 cdn-trust.com 的通信，无法感知真实目的地。
  • 主要应用：
    • C2 隐藏：Cobalt Strike 等工具利用域前置将 Beacon 流量伪装为合法 CDN 流量，规避检测和域名封禁。
    • 突破封锁：Signal、Telegram 等曾使用 Google App Engine 域前置绕过国家级封锁。
  • 现状与限制：
    • 主流云厂商已陆续禁止：AWS CloudFront（2018）、Google Cloud（2018）、Azure Front Door（2022）均强制校验 SNI 与 Host 一致性。
    • 替代技术：利用 Cloudflare Workers、腾讯云 EdgeOne 等边缘计算平台实现中继转发，SNI 指向合法边缘域名，Worker 再将请求转发至真实 C2。
  • 检测思路：
    • 解密流量或通过 TLS 检查，对比 SNI 与 Host 头是否不一致。
    • 监控去往已知 CDN IP 段的大流量，且非业务相关域名。
    • 分析 TLS 指纹（JA3/JA4）、周期性 Beacon 行为等异常模式。

29、钓鱼邮件的免杀怎么去做

  • 发件人认证绕过：
    • SPF 软失败：挑选 SPF 记录为 ~all 或机制包含大范围 IP 段的域名发信，或被伪造的域没有 SPF 记录。
    • DKIM 弱配置：部分域名 DKIM 密钥短于 1024 位可被爆破，或未完全启用 DKIM 签名，头信息篡改后不影响投递。
    • DMARC 宽松：当收件方域名的 DMARC 策略为 none 时，几乎可以随意伪造显示名称和 From 头，结合近似域名（如 rnicrosoft.com）提高可信度。
  • 邮件内容免杀：

    • 规避语义引擎：避免“密码”“验证”“紧急”“账单”等高危词，改用温和措辞，模仿系统自动通知或内部流程。
    • 回复链伪装：构造多封嵌套邮件的回复链，将恶意链接或附件藏在看似正常的历史会话中，降低被反垃圾引擎判定为欺诈的概率。
  • 附件免杀：
    • Office 宏免杀：
      • VBA 混淆：变量随机化、字符串 Chr() 拆分拼接、利用 CallByName 反射调用方法，避免特征码匹配。
      • VBA Stomping：生成带有宏的文档后，擦除压缩包中 vbaProject.bin 的 P-Code 源码，保留编译后的代码，静态查杀失效，打开时 Office 会自动重新编译执行。

      • 执行方式多样：不使用敏感的 CreateObject("Wscript.Shell")，改用 Shell.Application、WMI、MSHTA 或直接调用 Win32_Process 启动 payload。
    • 可执行文件/脚本免杀：
      • 白利用：利用系统白名单程序如 msbuild.exe 编译执行内联恶意 C# 代码，regsvr32.exe 执行远程 SCT 脚本，rundll32.exe 调用恶意 DLL 导出函数。

      • HTA/JScript 免杀：利用 ScriptControl 对象执行混淆后的 VBScript，或使用 DotNetToJScript 将 .NET 程序集转成单行 JScript，结合 MSHTA 执行。
  • 恶意链接免杀：
    • 信任跳板：使用 Google AMP 重定向、抖音/微博短链、腾讯动态域名跳转等合法平台，绕开 URL 黑名单检测。
    • 钓鱼页反沙箱：页面判断 User-Agent 是否为浏览器、检查 navigator.webdriver 是否开启、要求鼠标滑动解锁后才展示登录框，自动化分析工具无法触及表单。
    • 动态加载：初始链接返回无害内容或 404，只在检测到人工交互参数（如特定 Cookie、Referer）后才返回钓鱼页面。
  • 反沙箱/反分析通用手段：
    • 环境感知：检查 CPU 核心数、内存大小、屏幕分辨率、最近打开的文件，过滤掉虚拟机和沙箱。
    • 定时炸弹：代码中设定仅在某个时间范围或特定域名解析成功时才执行恶意逻辑。
    • 交互触发：要求点击“查看详情”、“加载图片”等二次交互，才下载真正 payload 或跳转到钓鱼页。
  • 多阶段投递与加密信道：
    • 无附件前置探测：先发一封无害邮件测试互动，对回复的地址投送恶意内容。
    • HTTPS + 域前置：payload 下载链接与 C2 通信均走 HTTPS，并借助高信誉 CDN（或边缘函数中继）隐藏后端域名，沙箱和网关只能看到合法 IP 和 SNI。

31、PHP反序列化

简述

把对象变成一个可以传输的字符串,为了方便传输.PHP文件在执行结束后会销毁对象,但有时候可能会用到销毁的了的对象,重复调用代码比较麻烦,于是有了序列化和反序列化,可以吧一个实例化对象长久的存在计算机磁盘上,调用的时候拿出来反序列化即可

漏洞原理

反序列化内容用户可控,且后台不正当地使用了魔术方法,使用户可以构造一个恶意的序列化字符串

利用流程

拿到代码,确定unserialize函数参数可控,然后看这些代码有哪些类,类中有哪些魔术方法,可以随着对象的创建或者销毁等操作调用哪些魔术方法,看魔术方法中有没有可以让我们利用的点,看类中有哪些参数可以通过反序列化修改,结合这些可以修改的参数和魔术方法中的利用点进行反序列化漏洞的利用

32、XSS/CSRF/SSRF漏洞原理与区别

XSS：跨站脚本攻击；CSRF：跨站请求伪造攻击；SSRF：服务器请求伪造攻击。

区别：
●XSS是服务器对用户输入的数据没有进行足够的过滤，导致客户端浏览器在渲染服务器返回的html页面时，出现了预期值之外的脚本语句被执行。
●DOM型:取出和执行恶意代码由浏览器端完成,通过js改变html代码
●反射型恶意代码存在URL里,存储型恶意代码在数据库里
●CSRF：CSRF是服务器端没有对用户提交的数据进行随机值校验，且对http请求包内的refer字段校验不严，导致攻击者可以利用用户的Cookie信息伪造用户请求发送至服务器。
●SSRF：SSRF是服务器对 用户提供的可控URL过于信任，没有对攻击者提供的URL进行地址限制和足够的检测，导致攻击者可以以此为跳板攻击内网或其他服务器。

XXE漏洞了解吗？

XXE漏洞即xml外部实体注入漏洞,发生在应用程序解析XML输入时，没有禁止外部实体的加载，导致可加载恶意外部文件，造成文件读取、命令执行、内网端口、攻击内网网站、发起dos攻击等危害。

43、金银票据

黄金票据（Golden Ticket）

黄金票据就是伪造krbtgt用户的TGT票据，krbtgt用户是域控中用来管理发放票据的用户，拥有了该用户的权限，就可以伪造系统中的任意用户

利用前提

拿到域控(没错就是拿到域控QAQ),适合做权限维持有krbtgt用户的hash值(aeshash ntlmhash等都可以,后面指定一下算法就行了)

防御

  • 限制域管理员登录到除域控制器和少数管理服务器以外的任何其他计算机（不要让其他管理员登录到这些服务器）将所有其他权限委派给自定义管理员组。这大大降低了攻击者访问域控制器的Active Directory的ntds.dit。如果攻击者无法访问AD数据库（ntds.dit文件），则无法获取到KRBTGT帐户密码
  • 禁用KRBTGT帐户，并保存当前的密码以及以前的密码。KRBTGT密码哈希用于在Kerberos票据上签署PAC并对TGT（身份验证票据）进行加密。如果使用不同的密钥（密码）对证书进行签名和加密，则DC（KDC）通过检查KRBTGT以前的密码来验证
  • 建议定期更改KRBTGT密码（毕竟这是一个管理员帐户）。更改一次，然后让AD备份，并在12到24小时后再次更改它。这个过程应该对系统环境没有影响。这个过程应该是确保KRBTGT密码每年至少更改一次的标准方法
  • 一旦攻击者获得了KRBTGT帐号密码哈希的访问权限，就可以随意创建黄金票据。通过快速更改KRBTGT密码两次，使任何现有的黄金票据（以及所有活动的Kerberos票据）失效。这将使所有Kerberos票据无效，并消除攻击者使用其KRBTGT创建有效金票的能力

白银票据（Silver Ticket）

黄金票据是伪造TGT（门票发放票），而白银票据则是伪造ST（门票），这样的好处是门票不会经过KDC，从而更加隐蔽，但是伪造的门票只对部分服务起作用,如cifs（文件共享服务），mssql，winrm（windows远程管理），DNS等等结语

简单了解黄金票据和白银票据：

黄金票据：是直接抓取域控中ktbtgt账号的hash，来在client端生成一个TGT票据，那么该票据是针对所有机器的所有服务。白银票据：实际就是在抓取到了域控服务hash的情况下，在client端以一个普通域用户的身份生成TGS票据，并且是针对于某个机器上的某个服务的，生成的白银票据,只能访问指定的target机器中指定的服务。

Windows

在 Linux 系统中，可以通过以下方式查看登录日志：

46、服务器存在 webshell，如何处理？

Webshell是一种Web服务器上的恶意代码，可以使攻击者远程控制服务器，并执行各种命令、上传下载
文件等操作。以下是处理服务器存在Webshell的一些方法：

  • 扫描和确认Webshell
    可以通过查看Web服务器日志或使用安全扫描工具来确认是否存在Webshell。找出所有可疑文件并进行彻底检查，以确定是否存在Webshell。
  • 删除Webshell
    如果确定存在Webshell，需要立即将其删除，并对系统进行全面清理。删除Webshell时，应同时清除与Webshell相关的文件和目录，并且应用程序代码也需要进行更新和修复。
  • 加强安全防护措施
    为了避免服务器再次被感染，需要加强安全防护措施，包括加强服务器的监控和日志审计、安装杀毒软件并定期更新、限制非管理员用户权限等。
  • 修改密码
    修改服务器管理员的账户密码，并确保强密码策略得到实施，建议使用包含大写字母、小写字母、数字和特殊字符等不同类型字符组成的复杂密码，并设置密码长度要求。
  • 更新和修复应用程序代码
    Webshell通常是由于应用程序存在漏洞或安全问题而被植入的。因此，为了防止类似问题再次发生，需要及时更新并修复应用程序代码中的安全漏洞。
    总之，在处理服务器存在Webshell的问题时，需要快速准确地识别和清除Webshell，并加强服务器和应用程序的安全防护措施。同时，也需要定期进行巡检和审计，及时发现和排除漏洞，以确保系统的完整性和安全性。

55、Linux 的 Selinux

SELinux（Security-Enhanced Linux）是一个针对 Linux 内核的安全模块，它可以限制进程和用户的访问权限，并提供更加细粒度的访问控制。SELinux 基于强制访问控制（MAC）模型，将每个进程/对象分配到不同的安全上下文中，并通过策略文件来定义这些上下文之间的关系。

在 Linux 中，SELinux 可以通过以下步骤进行设置：

  • 检查 SELinux 状态：使用命令 getenforce 或者 sestatus 来检查当前 SELinux 的状态，例如 enforcing、permissive 或者 disabled 等。
  • 修改 SELinux 配置文件 /etc/selinux/config：可以修改 SELINUX 参数的值为 enforcing（强制模式）、permissive（宽容模式）或者disabled（禁用 SELinux）等。
  • 安装和管理 SELinux 策略包：使用 yum 命令安装和管理 SELinux 相关的策略包，例如
    policycoreutils 和 selinux-policy 等。
  • 设定 SELinux 上下文：使用 chcon、semanage 和 restorecon 等命令来更改文件或目录的安全上下文。
  • 确认 SELinux 日志：在排除 SELinux 相关问题时可以使用命令 ausearch、ausearch、auditctl 等来确认 SELinux 日志。
    需要注意的是，在对 SELinux 进行设置和管理时，需要有一定的 Linux 系统管理经验和 SELinux 相关知识。因为错误的配置可能会导致系统不稳定或者无法启动，所以在操作前应仔细阅读相关文档并进行备份。

58、DDOS CC等攻击应急思路以及如何防范

DDoS（分布式拒绝服务攻击）和 CC（HTTP）攻击是目前比较普遍的网络攻击方式之一。以下是应急响
应思路以及如何防范这些攻击的建议：

  • 应急响应思路
    在受到DDoS和CC攻击时，应采取以下措施来降低攻击对系统的影响：
    快速检测并确认攻击类型和攻击源；
    提高带宽和资源利用率，以提供更强的承载能力；
    实施流量清洗、流量限制和IP封堵等措施，清除恶意攻击流量；
    配合 ISP 进行攻击源 IP 的追踪和协助；
    在攻击持续期间，保持实时监控和跟进，随时更新应急响应策略。
  • 如何防范
    为了预防DDoS和CC攻击，可以采取以下方法：
    增加网络带宽，并提高硬件设备的承载能力，以抵御较小规模的攻击；

实现DDoS和CC攻击的检测和防御机制，如基于流量和行为的检测技术和防御技术；
确保服务器和网站软件及时更新，以弥补已知漏洞；
实施流量清洗、流量限制和IP封堵等措施；
配置合理的防火墙规则和入侵检测系统（IDS）等安全设备；
加强网络安全教育，提高用户密码安全性，并定期对敏感数据进行备份和加密。
总之，在防范DDoS和CC攻击方面，需要综合考虑多种因素，包括增加带宽、提高硬件承载能力、实现攻击检测和防御机制、确保服务器和网站软件及时更新、加强网络安全教育等。同时，应该建立完善的应急响应机制和紧急处理预案，以做好应急响应准备和快速响应。

59、挖矿病毒判断以及处理方式

挖矿病毒是指利用受感染的计算机进行加密货币挖矿的恶意软件。以下是判断和处理挖矿病毒的一些方法：

  • 判断挖矿病毒
    通过以下迹象可以判断计算机是否被感染了挖矿病毒：
    计算机运行速度变慢，CPU占用率高；
    电脑风扇声音变大，温度升高；
    安全软件报告未知病毒或恶意软件；
    系统出现闪退或死机等异常情况。
  • 处理方式
    如果怀疑计算机已经被感染了挖矿病毒，可以采取以下措施：
    及时更新杀毒软件，并开启实时保护模式；
    扫描整个系统，清除所有恶意文件和注册表项；
    禁止不明程序自启动，及时关闭被感染的进程；
    在安装软件时仔细阅读许可协议，禁止安装未知来源的软件；
    加强账户权限管理，限制非管理员用户对系统的操作权限；
    及时备份重要文件，以便在恶意软件攻击后恢复数据。
    总之，在处理挖矿病毒方面，需要及时发现和清除感染，并加强系统的安全防护措施，如升级杀毒软件、限制非管理员用户权限、备份重要文件等。同时，也需要加强网络安全教育，提高用户对于计算机安全的意识和防范能力

60、同源策略是什么

同源策略（Same Origin Policy）是一种Web安全策略，它是浏览器中的一项重要特性，用于限制从一个源加载的文档或脚本如何与来自另一个源的资源进行交互。同源指的是三个关键元素：协议、主机和端口号，只有这三者完全相同的两个URL才被认为是同源的。
同源策略可以防止恶意网站访问其他网站的敏感数据，例如Cookie、LocalStorage和请求响应等信息，从而保护用户的隐私和安全。
例如，假设您正在访问一个银行的网站，该银行使用Cookie来存储您的登录凭据。如果攻击者能够在他们的网站上注入一些JavaScript代码并通过某种方式将其转移到您正在访问的银行网站上，那么该攻击者就可以访问您的Cookie，以便窃取您的个人信息。然而，由于同源策略的存在，攻击者无法访问您正在访问的银行网站的Cookie，因为它们不属于同一个源。
尽管同源策略对Web安全至关重要，但在某些情况下，它可能会成为开发过程中的挑战。为了解决这些问题，Web开发人员可以使用一些技术和标记，如跨域资源共享（CORS）和JSONP等来解决这些限制。
