一、判断一个页面是“Web页面”还是“API接口”，

最核心的区别在于：它是给人“看”的，还是给机器“读”的。

1. 肉眼直接看（最直观）

  • Web页面：返回的是图文并茂的界面，包含文字、按钮、图片、表格、导航栏，需要滚动浏览。
  • API接口：返回的通常是纯文本、一串乱码般的JSON数据（{ "code":200, "data":... }）、XML格式，或者干脆是空白页面加一行报错信息。没有任何CSS样式或图片装饰。

2. 查看浏览器地址栏（看URL后缀）

  • Web页面：URL通常以 .html、.htm、.php、.asp 结尾，或者干脆是纯路径（如 /home、/products/123），且不带奇怪的参数。
  • API接口：URL常包含明显的标识词，如 /api/、/v1/、/rest/、/graphql，或者后缀是 .json、.xml。例如 https://xxx.com/api/getUser?id=1。

3. 按 F12 打开开发者工具（最准确）

这是程序员最常用的方法：

  • 按 F12 或右键点击“检查”，切换到 “网络（Network）” 标签页，刷新当前页面。
  • 看当前页面的第一条请求记录（通常是第一个文件）：
    • 如果 类型（Type） 是 document（文档），且 响应头（Headers） 中的 Content-Type 是 text/html，说明是 Web页面。
    • 如果 类型 是 fetch、xhr，或者 Content-Type 是 application/json、text/plain，说明是 API接口。

4. 复制链接到新窗口打开（排除干扰）

如果当前页面看起来像数据，但带有样式（可能是调试工具），你可以直接复制当前页面的完整URL，打开一个无痕/隐私浏览窗口粘贴访问：

  • 如果看到漂亮的界面 → Web页面。
  • 如果打开后直接显示一堆大括号/方括号的数据 → API接口。

5. 看“页面”是否能交互（点击测试）

  • Web页面：点击链接会跳转、按钮有点击反馈、表单可以输入文字。
  • API接口：页面上没有任何可点击的按钮（除非是Swagger/Postman风格的API文档工具），点击空白处无任何反应，通常只能看到只读的文本数据。

一个特例需要注意：
如果你看到一个排版美观、带侧边栏、有“Try it out”按钮的页面，里面展示着JSON数据，这其实是API调试文档（如Swagger UI、Redoc）。它本身是“Web页面”，但它的核心用途是描述和测试“API接口”。你可以按上面的“维度3”查看Network类型来最终确认（它加载的是 text/html 文档）。

二、什么是接口

1. 核心比喻：接口就是“服务员”

想象你去一家高级餐厅吃饭：

  • 你（客户端）：想要吃饭，但不会做菜。
  • 后厨（服务器/数据库）：有所有食材和厨艺，能做菜，但不能直接出来跟你说话。
  • Web页面（前端界面）：就是餐厅的菜单和餐具，是给你看的、用来操作的东西。
  • 接口（API）：就是那个传菜的服务员。

接口的作用就是“传话”：你告诉服务员“我要一份宫保鸡丁”（发送请求），服务员把单子递给后厨，后厨做好菜后，服务员再把菜端到你面前（返回数据）。整个过程，你不需要知道后厨用的是铁锅还是砂锅，也不需要亲自去拿食材——服务员（接口）帮你搞定了所有沟通。

2. 接口到底长什么样？（技术视角）

在技术上，接口就是一套约定好的“暗号”，通常由以下四部分组成：

  1. 地址（URL）：就像餐厅的“传菜窗口”位置，比如 https://api.xxx.com/order。
  2. 方法（Method）：告诉你要干什么事。主要有四种：
    • GET（点菜）：只查看数据，不修改数据。
    • POST（下单）：提交新数据（比如注册账号、提交订单）。
    • PUT/PATCH（修改）：更新现有数据（比如修改昵称）。
    • DELETE（退菜）：删除数据。
  3. 参数（Params/Body）：你具体要什么东西。比如 {"dish": "宫保鸡丁", "spicy": "微辣"}。
  4. 返回结果（Response）：后厨给你的结果。通常是 JSON 格式的文本，比如 {"code": 200, "data": {"status": "已做好的菜"}, "msg": "成功"}。

3. 接口的三大核心特性（记住这三点）

  • 看不见（无界面）：接口没有按钮、没有颜色、没有排版，它就是后台的一串数据流。所以你刚才问“怎么判断页面还是接口”，就是因为接口没有界面可看。
  • 标准统一：不管你是苹果手机、安卓手机，还是电脑网页，只要按照这套“暗号”去请求，拿到的数据都是一模一样的。
  • 只传数据，不负责显示：接口只把“数据”（比如用户姓名、价格、列表）扔给你，至于这些数据在屏幕上是大字体还是小字体、是红色还是蓝色，那是“Web页面”负责排版的事。

4. 你生活中每天都在用接口

  • 查天气：你打开天气App，App调用天气接口（GET /weather?city=香港），接口返回 {"temp":28, "rain": false}，App再把“28°C”画成太阳图标给你看。
  • 微信支付扫二维码：你扫码这个动作，是向微信支付接口发送了“扣款请求”，接口返回“扣款成功”或“余额不足”，页面上才弹出弹窗提醒你。

5. 顺便帮你理清一个常有的误解

很多人以为“Swagger”或者“Postman”那种界面就是接口，其实那不是接口本身，那叫接口文档/调试工具——相当于服务员手里的“点菜平板”，只是用来帮你快捷填写“暗号”的工具，真正的接口运转在后厨（服务器）里。

所以

  • Web页面 = 你手中的菜单（给人看样式）。
  • API接口 = 那个传菜的服务员（给机器传数据）。

三、前后端是怎么通过接口联调的

前后端联调就像是服务员（前端）和厨师（后端）配合着把一道菜完美端给顾客的过程。

但在现实中，前端和后端往往是同时开工的。如果非要等厨师把菜全做完，服务员再开始写菜单，项目周期就太长了。所以，联调本质上是一场“按图纸对接”的协作游戏。

我用四个阶段，把前后端联调的全过程给你拆解透：

第一阶段：定“菜谱”（约定接口文档）

在开工前，前后端会坐在一起（或者用Swagger/YApi等工具），把所有的“暗号”提前定死。这份文档就是契约，包含：

  • 接口地址（URL）：比如 /getUserInfo。
  • 请求方式（GET/POST）。
  • 需要传什么参数（叫 userId 还是 uid？是数字还是字符串？）。
  • 会返回什么数据（返回的JSON里，用户名是叫 name 还是 userName？）。

现实中的坑：如果文档里写返回 userName，后端写代码时写成了 username，前端页面就会显示“undefined”（未定义）。联调最常见的 Bug，就是没按“菜谱”放调料。

第二阶段：各自“练手”（并行开发）

有了“菜谱”，两边就可以互不干扰地同时干活了：

  • 后端（厨师）：按照文档，在服务器上写代码，提供真实的接口数据。
  • 前端（服务员）：绝对不等着后端！ 前端会使用“Mock（模拟）工具”，自己造一份和文档格式一模一样的假数据。页面直接调用这份假数据，把按钮样式、排版动画全部做好。

这时候，前端页面看起来是跑通的，但数据全是假的——这叫“前后端分离开发”。

第三阶段：正式“对菜”（环境连接）

当两边都觉得自己写好了，就进入最核心的联调阶段：

  1. 前端切换“靶子”：前端把代码里的接口地址，从“本地假数据地址”一键切换成“后端真实服务器的地址（如 http://192.168.1.xxx:8080）”。
  2. 发起“实弹请求”：前端点击页面上的“登录”按钮，请求真实发送到了后端服务器。
  3. 后端接收并响应：后端收到请求，去数据库查数据，把结果（JSON格式）返回给前端。
  4. 前端渲染：前端拿到真实数据，刷新页面。

如果页面显示正常，代表这条路“通”了；如果页面报错或乱码，就进入激烈的“调试阶段”。

第四阶段：拿着“工具”找 Bug（调试与排查）

联调通常不是一次过的，出问题时会用到下面这些“破案工具”：

  • 前端查看网络请求（F12 -> 网络（Network））：前端按F12，看请求有没有发出去？返回的HTTP状态码是什么？
    • 如果是 404：说明地址写错了（走错厨房了）。
    • 如果是 500：说明后端代码崩了（厨师锅烧糊了）。
    • 如果是 400：说明参数传错了（把“微辣”写成了“巨辣”）。
  • 后端查看日志（Log）：后端看服务器控制台，看前端传过来的参数到底是什么。如果发现没收到参数，就是前端没传对；如果收到参数但查不到数据，就是数据库问题。
  • 使用接口调试工具（Postman/Apifox）：为了判断到底是谁的错，后端会把接口地址复制到Postman里自己调一下。如果Postman里数据正常，但前端页面报错——那就是前端解析的代码写错了；如果Postman里也报错——那就是后端的锅，赶紧去改。

一个联调时的真实“撕逼”场景（帮你避坑）

前端（指着屏幕）：“后端大哥，我这边报错了，用户信息没出来。”
后端（查日志）：“我明明返回了数据啊，你看 {"data":"null"}。”
前端：“你文档说返回 null，但我代码里是按 []（空数组）写的循环，你让我怎么显示？！”
（解决方案：按文档严格对齐，或者后端改成 []，前端改成判空处理）

最终阶段：提测与上线

所有接口都走通后，前后端会一起把代码合并，部署到测试环境给测试人员测。测试通过后，再一起部署到线上生产环境，正式对外服务。

最后送你一句老程序员的“保命心法”：

联调之前，打死不认是自己的Bug；联调之后，打开抓包工具（F12），看数据包到底长啥样——数据包不会说谎，它是唯一的法官。

四、跨域（CORS）是什么

跨域（CORS）堪称前后端联调时的“头号拦路虎”。联调时页面报错，十有八九就是它搞的鬼。

接住我们之前“餐厅”的比喻，跨域就是“保安查工作证”。我带你三分钟彻底吃透它。

1. 到底什么是“跨域”？（保安拦人）

浏览器（保安）有一个极其严格的安全规则——同源策略。
“同源”指的是三个完全一样：协议（http/https）、域名（xxx.com）、端口号（:8080）。

现实场景：
你的前端页面跑在 http://localhost:3000（你在A餐厅当服务员），但你要调用的后端接口在 http://api.xxx.com:8080（B餐厅后厨）。

  • 浏览器保安一看：“不对啊，你的工牌是A店的，凭什么去B店后厨端菜？拦住！”
  • 于是，浏览器直接报错（控制台红字）：Access-Control-Allow-Origin 错误。注意：请求其实发出去了，后端也处理了，只是浏览器扣住了返回的数据，不交给前端代码。

一个关键认知：跨域报错，是浏览器的锅，不是后端的锅，更不是前端的锅。后端接口用Postman调是好的，一放浏览器就报错——就是因为Postman没保安管。

2. 跨域是怎么“打架”的？（简单请求 vs 预检）

为了安全，浏览器会把请求分两类，采取不同策略：

  • 简单请求（如 GET 查数据）：浏览器直接发请求，如果后端没在返回头里写“允许A店进入”，浏览器就直接把数据丢进垃圾桶，前端拿不到。
  • 复杂请求（如 PUT 修改数据，或带自定义头）：浏览器更谨慎，正式请求前，先发一个 “预检请求（OPTIONS）”。这就好比保安先拦着你问：“喂，B店后厨，A店服务员想进来端菜，你允许吗？”只有B店后厨回复“允许（Access-Control-Allow-Origin）”，浏览器才放行正式的请求。

3. 怎么解决跨域？（三种硬核方案）

既然拦路虎是浏览器，我们就要给它看“通行证”。

方案一：后端加“白名单”（最正统、最推荐）

这是最终的线上解决方案。后端开发在代码里添加一个响应头（Response Header），明确告诉浏览器：“虽然前端是外来的，但我信任它，允许它拿数据。”

如果遇到复杂请求，后端还得加上 Access-Control-Allow-Methods（允许的方法）和 Access-Control-Allow-Headers（允许的头信息）。

方案二：前端“走后门”代理（联调神器，无需后端改代码）

在前后端联调开发阶段，前端可以用自己的服务器当个“二传手”。

  • 原理：服务器和服务器之间通信，不存在跨域限制（只有浏览器有保安）。
  • 操作（以Vue/React为例）：在配置文件中设置 proxy（代理）。前端请求先发给自己的前端服务器（比如 localhost:3000），前端服务器再把请求转发给后端服务器（api.xxx.com），拿到数据后丢给浏览器。
  • 因为浏览器最终访问的是 localhost:3000（同源），保安就不管了。
方案三：JSONP（古董方案，仅限 GET）

利用 <script> 标签不受跨域限制的漏洞来加载数据。但只能发 GET 请求，且不安全，现在极少使用，除非你在维护10年前的老项目。

4. 联调时的“终极裁判”操作

如果在联调时遇到跨域报错，作为前端，你可以按 F12 -> 网络（Network），看两条记录：

  1. 先看 OPTIONS 请求（如果有的话）：如果它返回状态码是 404 或 500，说明后端根本没给预检请求写路由，直接让后端加上。
  2. 再看真正的请求（GET/POST）：如果它状态码是 200，但控制台依然报跨域，说明后端没加 Access-Control-Allow-Origin 响应头。这时候直接把截图甩给后端：“大哥，请你响应头里给我开一下白名单。”

一句话总结：
跨域就是浏览器怕你被钓鱼，强行搞的安检。上线了就让后端开白名单，开发调试就用前端代理（Proxy）。只要记住“浏览器拦、服务器解”这六个字，以后联调遇到它，你就能冷静地找对责任人。
