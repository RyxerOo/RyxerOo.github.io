    • 1、拿到一个待检测的站，你觉得应该先做什么？

    收集信息：whois、网站源IP、旁站、C段网站、服务器系统版本、容器版本、程序版本、数据库类型、二级域名、防火墙、维护者信息另说…

    2.mysql的网站注入，5.0以上和5.0以下有什么区别？

    5.0以下没有information_schema这个系统表，无法列表名等，只能暴力跑表名；5.0以下是多用户单操作，5.0以上是多用户多操做。

    3.在渗透过程中，收集目标站注册人邮箱对我们有什么价值？

    (1) 丢社工库里看看有没有泄露密码，然后尝试用泄露的密码进行登录后台
    (2) 用邮箱做关键词进行丢进搜索引擎
    (3) 利用搜索到的关联信息找出其他邮箱进而得到常用社交账号
    (4) 社工找出社交账号，里面或许会找出管理员设置密码的习惯
    (5) 利用已有信息生成专用字典

    4.判断出网站的CMS对渗透有什么意义？

    查找网上已曝光的程序漏洞，如果开源，还能下载相对应的源码进行代码审计。

    5.一个成熟并且相对安全的CMS，渗透时扫目录的意义？

    (1)敏感文件、二级目录扫描
    (2)站长的误操作比如：网站备份的压缩文件、说明.txt、二级目录可能存放着其他站点

    45.nmap扫描的几种方式？

    Nmap 的扫描方式可以从端口扫描技术、服务探测和主机发现等层面来分类。面试中最常问的是端口扫描技术，主要有以下这些，每种都通过不同标志位组合来推断端口状态，绕过检测的能力也不同：

    • TCP SYN 扫描（-sS）：默认扫描方式，发送 SYN 包后若收到 SYN/ACK 则端口开放，收到 RST 则关闭。它不完成三次握手（半开扫描），速度快、隐蔽性较好，但需要 root 权限。
    • TCP connect 扫描（-sT）：调用系统 connect() 完成完整 TCP 连接，扫描器会按正常流程建立连接然后断开。无需特殊权限，但会在目标应用日志中留下大量连接记录，隐蔽性差。
    • UDP 扫描（-sU）：发送 UDP 包，通过 ICMP 端口不可达或应用响应来判断端口状态。UDP 无连接特性导致扫描速度慢、结果易受网络环境影响，但对发现 DNS、SNMP 等 UDP 服务必不可少。
    • TCP ACK 扫描（-sA）：只设置 ACK 标志位，用于探测防火墙规则而不是端口开关。如果收到 RST 说明端口未被防火墙过滤；无响应或收到 ICMP 不可达则为被过滤状态。
    • TCP FIN / NULL / Xmas 扫描（-sF / -sN / -sX）：分别发送 FIN、无标志、或 FIN+PSH+URG 标志包。依据 RFC 793，开放端口对这类异常包会忽略（不响应），关闭端口会返回 RST。这类扫描可绕开一些无状态防火墙和仅监视 SYN 的 IDS。
    • TCP Window 扫描（-sW）：检查返回 RST 包的 TCP 窗口大小，某些系统开放端口和关闭端口的窗口值不同，可用于辅助判断。
    • TCP Maimon 扫描（-sM）：发送 FIN/ACK 包，探测方法类似 FIN 扫描，部分 BSD 系统会有不同响应。
    • IP 协议扫描（-sO）：确定目标主机支持哪些 IP 协议（如 ICMP、TCP、IGMP 等），不是扫描端口，而是扫描协议号。
    • FTP bounce 扫描（-b）：利用老旧 FTP 服务器的代理功能，通过 PORT 命令让 FTP 服务器帮你扫描目标，可隐藏真实扫描源，现已很少用。

    除了上述端口级别的技术，Nmap 还提供高层探测方式：

    • 版本扫描（-sV）：向开放端口发送多种探测包，尝试识别服务名称和具体版本。
    • OS 检测（-O）：通过分析 TCP/IP 指纹（如初始序列号、选项排列等）识别目标操作系统。
    • 脚本扫描（-sC 或 --script）：调用 NSE 脚本引擎，进行漏洞发现、弱口令检测、信息收集等高级功能。
    • 空闲扫描（-sI zombie_host）：利用一台“僵尸”主机的 IPID 增长规律，完全隐藏自身 IP 进行端口扫描，隐蔽性最高。

    65.gpc 魔术引号？

    GPC 魔术引号是 PHP 早期版本中一个已被废弃的安全机制。

    GPC 是 GET、POST、Cookie 三种输入方式的缩写，魔术引号指的是 magic_quotes_gpc 这个配置项。当它开启时，PHP 会自动在这些外部输入的数据中，给单引号、双引号、反斜杠和 NULL 字符前面加上反斜杠，也就是执行类似 addslashes() 的操作。

    它最初的目的是想帮开发者自动防范 SQL 注入，但实际效果适得其反，后来被公认为是一个失败的设计。原因主要有三点：
    第一，污染原始数据。它不区分上下文，对所有输入都加转义，哪怕你只是把数据显示在 HTML 里或者存到文本文件里，也会多出不必要的反斜杠，开发者还得反过来用 stripslashes() 清除。
    第二，带来虚假的安全感。它只转义了部分字符，对数字型注入完全无效，也不能防御其他类型的注入。很多开发者因此认为数据已经‘安全’了，反而忽略了真正的防御措施。
    第三，可移植性极差。它是依赖 php.ini 配置的开关，同一套代码在不同服务器上表现可能完全不同，于是大量遗留代码里充斥着 if (get_magic_quotes_gpc()) 这样的兼容写法。

    基于这些问题，PHP 在 5.3 里将它标记为弃用，并在 5.4 版本彻底移除。现在正确的做法，是绝对不要依赖任何自动转义，必须使用 PDO 预处理语句 或 MySQLi 预处理语句，通过参数绑定把 SQL 逻辑和数据完全分离，这才是根本防御 SQL 注入的方式。如果在项目中看到相关代码，那就意味着项目还跑在极老旧的 PHP 版本上，是需要尽快重构掉的。

    怎么重构掉

    重构掉对魔术引号的依赖，首先要确保 PHP 版本已升级到 5.4 以上，然后全局搜索并彻底删除所有 get_magic_quotes_gpc 判断及对应的 stripslashes 反转义代码，让输入数据保持原始状态；接着最关键的是把项目里所有靠拼接变量形成的 SQL 语句全部改为 PDO 或 MySQLi 的参数化预处理语句，从根本上防范注入，而不是依赖任何自动转义；同时还要把其他场景中滥用的 addslashes 替换为对应上下文的正确函数，比如输出 HTML 时用 htmlspecialchars、生成 JSON 时用 json_encode；如果历史数据库里确实存储了带反斜杠的脏数据，还需要写一次性脚本把多余的 \'、\" 等清理掉

      168、DDoS 攻击了解多少

      DDoS（分布式拒绝服务）是通过大量分布式的受控主机（僵尸网络）向目标发起海量请求，耗尽目标网络带宽、系统资源或应用服务处理能力，致使正常用户无法访问。

      常见攻击类型与原理：

      1. 流量耗尽型（带宽攻击）
        • UDP Flood：发送大量UDP包到随机端口，目标忙于处理无用数据包和回复ICMP。
        • ICMP Flood / Ping Flood：海量ICMP请求耗尽带宽。
        • DNS/NTP/Memcached 放大攻击：利用协议特性，伪造目标IP向开放服务器发送小请求，服务器返回数倍大的响应，反射到目标。
      2. 协议攻击型
        • SYN Flood：发送大量伪造源IP的SYN包，占满TCP半连接队列，使服务器无法接受新连接。
        • Ping of Death：发送超大或畸形的ICMP包导致系统崩溃。
        • Smurf 攻击：伪造目标IP向广播地址发ICMP，大量主机回复淹没目标。
      3. 应用层攻击
        • HTTP Flood / CC攻击：看似正常的HTTP请求，但大量请求动态页面、搜索接口等消耗数据库资源。
        • Slowloris：使用极低速率维护大量HTTP连接，耗尽服务器的并发连接池。
        • Slow Read / RUDY：类似低速传输，使连接长时间不释放。

      防御思路：

      • 流量清洗与黑洞路由，利用CDN和云高防IP分散/吸收攻击。
      • 部署专业Anti-DDoS硬件或服务，针对SYN Flood开启SYN Cookie。
      • 配置WAF和连接限制，如限制单一IP并发连接数、请求速率。
      • 协议强化：禁用不必要的服务，使用反向代理，优化服务器和应用程序性能。
      • 监控预警，设计弹性扩容架构。

    172、xss 如何盗取cookie？

    XSS 盗取 Cookie 的核心原理是：利用 JavaScript 在受害者浏览器中执行，通过 document.cookie 读取当前域名下的 Cookie，然后将数据发送到攻击者控制的服务器。

    具体攻击过程如下：

    （1）寻找注入点：攻击者在目标网站上找到 XSS 漏洞（反射型、存储型或 DOM 型），并构造恶意脚本。

    （2）构造盗取载荷：
    经典的攻击载荷通常只需一行代码，其作用是“获取 Cookie 并外传数据”。例如：

    • **document.cookie**：读取当前页面的所有（非 HttpOnly）Cookie。
    • **new Image().src**：一种常见的外传方式，浏览器会自动发起跨域 GET 请求，攻击者只需在服务器日志或 steal.php 中接收参数即可。其他方式还包括 fetch、XMLHttpRequest 等。

    （3）接收与利用：
    攻击者在自己的服务器（attacker.com）上监听请求，当受害者访问被注入脚本的页面时，其浏览器就会悄悄将 Cookie 作为参数发送给攻击者。攻击者拿到会话 ID（Session ID）后，将其设置到自己的浏览器中，即可直接冒充受害者登录账户。

    为什么能成功？关键前提：
    盗取成功的前提是目标 Cookie 没有设置 HttpOnly 标志。HttpOnly 是 Cookie 的一个安全属性，一旦设置，浏览器会禁止 JavaScript 访问该 Cookie，使其无法被 document.cookie 读取。这正是防御 XSS 盗取 Cookie 的核心手段之一。

    防御措施总结：

    • 开启 HttpOnly：特别是 Session Cookie，这是防范盗取的最直接手段。

    173、tcp、udp 的区别及tcp三次握手，syn攻击？

    （1）TCP与UDP的区别

    特性 TCP（传输控制协议） UDP（用户数据报协议）
    连接性 面向连接，通信前需建立连接 无连接，直接发送数据
    可靠性 可靠，通过确认、重传、序号保证无差错、不丢包、按序到达 不可靠，不保证送达，不保证顺序
    效率 开销较大，有流量控制和拥塞控制 开销小，无控制机制，传输速度快
    传输模式 面向字节流，数据无边界 面向报文，保留报文边界
    通信方式 仅支持一对一 支持一对一、一对多、多对多（广播多播）
    头部开销 至少20字节 仅8字节
    典型应用 HTTP、FTP、SSH、SMTP 等 DNS、视频流、VoIP、在线游戏

    （2）TCP三次握手过程

    三次握手是TCP建立可靠连接的必经步骤，核心是交换彼此的初始序列号并确认双方的收发能力。

    1. 第一次握手（客户端 → 服务器）
      客户端发送一个SYN包，指定自己的初始序列号 seq=x，并设置SYN标志位，进入 SYN_SENT 状态。
      （表达：“我要连你，我的序号是x”）
    2. 第二次握手（服务器 → 客户端）
      服务器收到SYN，回复一个SYN+ACK包，包含自己的初始序列号 seq=y，同时对客户端的序列号进行确认 ack=x+1，进入 SYN_RCVD 状态。
      （表达：“收到，我的序号是y，我准备好收你的x+1了”）
    3. 第三次握手（客户端 → 服务器）
      客户端发送ACK包，确认号 ack=y+1，序号为 seq=x+1。双方进入 ESTABLISHED 状态，连接建立。
      （表达：“收到你的y，现在开始传数据”）

    为什么是三次而不是两次？ 主要是为了防止失效的SYN请求突然到达服务器而产生错误连接，保证双方都确认对方收发正常。

    （3）SYN攻击（SYN Flood）

    SYN攻击是一种典型的拒绝服务（DoS）攻击，利用TCP三次握手的机制进行资源耗尽。

    攻击原理：
    攻击者伪造大量不存在的源IP地址，向服务器发送SYN包。服务器收到后，会回复SYN+ACK并将该半连接存入半连接队列（SYN queue），等待客户端ACK。由于源IP是伪造的，ACK永远不会到来，服务器只能重发SYN+ACK并维持该半连接直到超时（默认可达几十秒）。当大量虚假SYN包瞬间涌来时，服务器的半连接队列会被迅速占满，无法再为任何新的正常连接提供服务。

    常见防御手段：

    • SYN Cookie：最有效的防御，服务器不立即为SYN分配资源，而是根据客户端IP、端口等信息加密生成一个Cookie放在第二次握手的ack序号中。只有收到第三次握手并验证Cookie有效后，才分配资源建立连接。这样攻击者的SYN不会消耗队列内存。
    • 增大半连接队列与缩短超时：一定程度上缓解，但无法根治。
    • 防火墙/IPS限制速率：对同一源IP的SYN速率做限制，但面对伪造分布式攻击效果有限。
    • 使用SYN Cache：只精简地缓存必要信息，减少资源占用。

    总结： TCP面向连接可靠，UDP无连接高效；三次握手确保连接建立；SYN攻击正是利用了握手过程中服务器必须保留半连接状态的弱点，而SYN Cookie是目前最核心的防御思路。

    180、什么是CC攻击？

    CC攻击（Challenge Collapsar，挑战黑洞）是一种应用层DDoS攻击，专门针对Web服务器，通过大量看似正常的HTTP请求耗尽服务器资源，导致正常用户无法访问。

    攻击原理

    CC攻击的目标不是带宽，而是服务器的计算资源（CPU、内存、数据库连接等）。攻击者控制大量代理或肉鸡，向目标网站反复请求那些最消耗资源的动态页面，例如：

    • 复杂的数据库查询页面
    • 搜索功能
    • 文件下载/上传接口
    • 动态生成图片或报表的页面

    这些请求完全符合HTTP协议，携带正常的User-Agent和Cookie，甚至模拟用户行为（如点击、登录），因此很难通过简单的IP封禁或协议检测来识别。当大量此类请求同时涌入，服务器的数据库连接池、PHP进程池很快耗尽，Web服务器变得极慢甚至宕机。

    与SYN Flood攻击的区别

    攻击类型 层级 目标 特点
    SYN Flood 网络层/传输层 消耗TCP连接队列、带宽 发送大量SYN包不完成握手
    CC攻击 应用层 消耗CPU、内存、数据库等后端资源 完成完整HTTP请求，模拟真实用户

    可以说，SYN攻击打的是服务器的“门卫”，CC攻击打的是服务器的“心脏”。

    防御手段

    因为CC攻击在应用层，防御需要结合多层策略：

    • 应用层WAF：设置每秒请求限制（Rate Limiting）、人机验证（JS挑战、验证码），识别异常高频IP。
    • CDN加速与缓存：将动态请求缓存为静态内容，尽可能让CDN节点直接响应，减轻源站压力。
    • 后端优化：优化慢查询、使用连接池、对高频接口做业务级别的限流（如每个用户每分钟只能搜索10次）。
    • 行为分析：利用机器学习识别异常流量模式，如是否来自同一客户端特征、是否总是请求同一高消耗URL。
    • 保留原始日志：便于事后溯源分析攻击源头特征。

    一句话总结：CC攻击就是攻击者用海量“正常”的HTTP请求，让目标服务器忙于处理复杂逻辑而无法响应正常用户，是Web服务器最常见的流量型应用层攻击之一。

    181、Web 服务器被入侵后，怎样进行排查？

    Web 服务器被入侵后的排查要遵循应急响应的流程，核心是找痕迹、定源头、清后门、做加固。我的排查思路大致分几步：

    第一步，立即隔离并保留现场。 先通过安全设备或主机防火墙对服务器进行网络隔离，避免横向扩散，但不要直接关机，否则内存和登录状态等易失数据会丢失。同时给系统做镜像或快照，备份日志，保证后续分析合法。

    第二步，检查异常连接和进程。 登录后先用 netstat -anp 看有没有可疑的对外连接或监听端口，尤其注意反弹 shell 类的 ESTABLISHED 连接。再用 ps auxf 或 top 查找高 CPU 的未知进程，关注进程名与系统相似但路径不同（如 /tmp/.bash）的情况，以及进程的 PPID 是否为 1 且无父进程的孤儿进程。还可以用 lsof 查看进程打开了哪些文件。

    第三步，检查账号与登录日志。 查看 /etc/passwd 和 /etc/shadow 有无新增的异常 UID 为 0 的账户；用 last、lastb、lastlog 分析登录记录，是否存在异地 IP、异常时间段登录；检查 /var/log/secure 或 /var/log/auth.log 中的登录认证失败/成功日志，以及 wtmp/btmp 二进制日志。

    第四步，排查持久化机制。 查看定时任务：crontab -l、/etc/crontab、/etc/cron.* 等目录；检查自启动脚本如 /etc/rc.local、/etc/init.d/ 下的可疑项；检查 systemd 的 service 文件；查看 ~/.ssh/authorized_keys 是否被写入攻击者公钥；搜索常见 webshell 文件特征，比如含有 eval、base64_decode、system 等函数的 php/jsp 文件，可结合 find 命令查找最近修改的文件。

    第五步，分析 Web 和数据库日志。 从 nginx/apache 的 access.log 中分析是否存在异常 URI、大量 404/500 或 payload 特征，比如 SQL 注入、XSS、webshell 上传的 POST 请求。可以借助日志分析工具或者简单的 awk、grep 统计高频 IP 和特定攻击 pattern。数据库日志也要检查是否有异常查询或导出操作。

    第六步，清理与恢复。 确定后门全部清除，漏洞修复后（如上传接口加权限校验、注入参数绑定），重新部署干净版本的程序，修改所有账号密码，并从备份恢复数据。最后还要复盘入侵原因，输出安全事件报告。

    常用工具方面，我还会用到 chkrootkit、rkhunter 查 rootkit，clamav 扫病毒，aide 做文件完整性校验，以及使用 auditd 审计可疑文件操作。

    182、dll 文件是什么意思，有什么用？DLL 劫持原理以及如何防止 DLL 劫持？

    DLL 是动态链接库（Dynamic Link Library）的缩写，它是 Windows 下的一种可执行文件格式，里面封装了函数、数据和资源，供其他程序在运行时动态调用，而不是直接编译到 exe 里。它的好处是代码共享、节省内存（多个进程可共享同一 DLL）、模块化更新（只替换 DLL 而不改主程序），比如 kernel32.dll、user32.dll 就提供了大量 API。

    DLL 劫持的原理，是利用 Windows 加载 DLL 时的搜索顺序机制。当一个程序需要加载某个 DLL 时，如果只给出文件名而没有完整路径，系统会按照特定顺序去一系列目录中查找。攻击者可以将恶意 DLL 放在搜索顺序中优先于系统目录的位置（例如程序当前目录），让程序误加载这个恶意 DLL，代替正常的那个。如果恶意 DLL 导出了和正版 DLL 同样的函数名，它就可以在程序调用这些函数时执行攻击代码，并转发调用到原始 DLL 保持功能正常。这常被用于绕过安全软件、提权或长期驻留。

    典型的劫持场景有：

    • 程序目录放置：将恶意 DLL 与正常的 exe 放在同一目录，双击运行，优先加载恶意 DLL。
    • 利用已知的 DLL 名称：比如针对特定程序缺失的 DLL 或版本问题，放置 version.dll、userenv.dll 等系统已知的劫持目标。
    • 搜索路径劫持：通过修改环境变量 PATH 或利用 System32 之前的目录顺序。

    如何防止 DLL 劫持：

    • 开发者角度：程序在加载 DLL 时使用绝对路径，或者调用 SetDllDirectory 和 LoadLibraryEx 并指定 LOAD_LIBRARY_SEARCH_SYSTEM32 等安全搜索标记，移除当前目录搜索。
    • 系统管理员：启用安全策略，如设置注册表 CWDIllegalInDllSearch 值为 1，禁止程序从当前工作目录加载 DLL；使用 AppLocker 或软件限制策略控制可加载的 DLL 路径。
    • 终端防护：安全软件监控异常的 DLL 加载行为，例如非可信路径的 DLL 被关键进程加载。
    • 日常维护：及时更新操作系统和应用，禁用不必要的服务，对文件目录权限进行最小化授权，防止攻击者写入恶意 DLL。

    183、Rootkit 是什么意思，怎么防范和排查？

    Rootkit 是一类恶意软件或工具集，它最核心的目的是隐藏。它通过篡改操作系统内核、驱动或应用程序，让攻击者能够持续获得超级用户权限，同时将自身的文件、进程、网络连接、注册表项等隐藏起来，使得常规的系统管理工具（如任务管理器、文件资源管理器）无法发现。

    Rootkit 主要分为几种：

    • 用户态 Rootkit：通过替换或 Hook 用户空间的库函数（如拦截 ps 命令调用的 readdir），实现隐藏。相对容易被完整性检查发现。
    • 内核态 Rootkit：最危险，直接修改内核数据结构（如系统调用表 SSDT）、插入内核模块（LKM）或驱动，能拦截内核行为，完全欺骗操作系统。比如隐藏进程只需从内核的进程链表中摘除条目。
    • 固件 Rootkit：隐藏在 BIOS、UEFI、硬盘固件等底层中，重装系统也无法清除。

    防范措施：

    • 开启安全启动（Secure Boot），确保引导链上的固件和系统内核未被篡改。
    • 保持系统与软件更新，及时打补丁，阻断漏洞利用提权。
    • 使用最小权限原则，限制 root/管理员权限的滥用，防止攻击者轻易安装内核模块。
    • 部署内核加固和完整性检查，如 Linux 的 SELinux/AppArmor 和 IMA/EVM，Windows 的驱动签名强制和 Virtualization-based Security (VBS)。
    • 定期使用可信的扫描工具检查，并与安全基线对比。

    排查方法：
    由于 Rootkit 会欺骗本地系统，所以排查需要借助外部可信环境，思路是交叉视图对比。

    • 离线挂载分析：将可疑磁盘挂载到干净系统，使用 chkrootkit、rkhunter、ClamAV 检查已知 Rootkit 特征。直接查看文件系统和目录，与系统运行时看到的文件对比，若 /proc 中不存在的进程却在磁盘有可执行文件，就很可疑。
    • 内存取证：使用 Volatility 等工具分析内存转储，恢复被隐藏的进程和网络连接，查找内核挂钩痕迹，如 SSDT Hook、IDT Hook 等。
    • 专用检测工具：例如微软的 RootkitRevealer（较老）、GMER、TDSSKiller 等针对特定 Rootkit 族。
    • 网络行为分析：从外部抓包或流分析，对比主机内查看的网络连接，若发现对外发包但本机 netstat 看不到连接，极可能是内核级 Rootkit 在隐藏。
    • 严格场景下重装：一旦确认存在内核级或固件 Rootkit，最稳妥的方案是擦除硬盘、重刷固件、从头重建系统，因为很难保证完全清理干净。

    总之，Rootkit 的排查核心是‘不信任被入侵机器的反馈’，必须从外部或底层视角检查。

    • 首先最需要注意的地方是系统的启动项，可以在“运行”-输入“msconfig命令” 在打开的系统配置实用程序里的启动列表查看，并且服务也要注意一下，如果对 电脑不是太熟悉的童鞋建议使用360安全卫士的开机加速功能，来查看有无异 常的可以启动项和服务项，因为在后门木马中99%都会注册自己为系统服务， 达到开机自启动的目的，如果发现可疑项直接打开相应的路径，找到程序文件， 直接删除并且禁止自启动；

    • 查看系统关键目录system32和系统安装目录Windows下的文件，xp系统 下两者默认路径分别是C:\WINDOWS\system32和C:\WINDOWS\。然后最新 修改的文件中有没有可疑的可执行文件或dll文件，这两个地方都是木马最喜欢 的藏身的地方了（小提示：一定要设置显示所有的文件的文件夹哦）。

    • 观察网络连接是否存在异常，还有“运行”-“cmd”-“netstat-an”查看有没有可疑 或非正常程序的网络连接，如果对电脑不是很熟悉建议大家使用360的流量监 控功能更加直观和方便，尤其注意一下远程连接的端口，如果有类似于8000等 端口就要注意了，8000是灰鸽子的默认端口，记得有一次自己就在后门木马测 试中在网络连接中发现8000端口，当然意思不是说只要没有8000端口的网络 连接就一定安全，因为8000端口只是灰鸽子上线的默认端口，并且端口是可以 更改的。

    185、主机被入侵，你会如何处理这件事自查解决方案

    如果主机被入侵，我会按“隔离研判、排查清理、加固恢复”三个阶段来处理：

    首先是隔离与研判。
    立即通过网络或主机防火墙隔离主机，但切忌关机，以保留内存和网络状态。同时，第一时间用 netstat -anp 分析所有网络连接，重点寻找对外部可疑 IP 的 ESTABLISHED 连接，这是发现反弹 Shell 或木马通信最直接的手段。若发现，立刻用 kill 终止进程。之后，将系统做快照，备份 /var/log 等关键日志到外部，作为后续溯源证据。

    然后是核心的排查与清理，这是自查的关键。
    我会进行“四层自查”：
    第一，账号安全。 检查 /etc/passwd 和 /etc/shadow，清除攻击者创建的隐藏账户（尤其是 UID 0 的）。用 last 和 lastb 分析登录记录，若发现异地登录，立即将所有弱密码改为“字母+数字+特殊符号”10位以上的强密码。同时检查 ~/.ssh/authorized_keys，删除攻击者植入的公钥。
    第二，木马与后门。 先用 ps auxf 和 netstat -anp 交叉对比，找到关联的可疑进程。对于常见 Linux 木马，可直接按特征清理：比如发现文件被用 chattr +i 锁定时，需先 chattr -i 再 rm -rf 删除，并注意用干净备份恢复 ps、netstat、lsof 等系统命令，防止被木马替换。尤其要扫描 /usr/bin/、/tmp 等目录，找到如 /proc/*/exe 指向的已删除但仍在运行的恶意进程，并用 killall -9 彻底终止。
    第三，应用与中间件漏洞。 这是入侵最常见的入口。立即扫描并修复 Web 应用及组件的已知漏洞，如 Struts 2 RCE、ElasticSearch 未授权等。同时，把 Jenkins、Tomcat、phpMyAdmin、WebLogic 等所有后台密码都提升至强密码强度。特别检查 MySQL、Redis、FTP 等服务的口令，尤其是 Redis 无密码导致的写 SSH 公钥漏洞，确认 /root/.ssh/ 下没有可疑文件，并将 Redis 改为强密码访问，如无公网需求，用 bind 127.0.0.1 禁止外部连接。
    第四，持久化后门。 全面排查 crontab、/etc/rc.local、systemd service 等自启动任务，删除恶意下载和执行脚本，确保木马无法“复活”。

    最后是加固与恢复。
    清除完成后，修改所有口令，从干净备份恢复数据和程序。开启 WAF 和云盾（安骑士）等安全防护产品的所有功能，进行全盘病毒查杀和实时入侵检测。最终如果排查极为困难，或存在内核级 Rootkit 无法根除的迹象，则采取最彻底方案：将数据下载备份、本地杀毒后，重置系统盘和数据盘，从零重建环境并部署应用，再进行一次完整加固和检测，确保环境绝对干净。
