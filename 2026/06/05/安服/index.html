1、如何发现SQL注入漏洞

发现SQL注入漏洞通常分为信息收集和漏洞探测两个阶段：

信息收集阶段，需要确定目标网站使用的数据库类型（MySQL、Oracle、PostgreSQL等），以及应用程序的所有输入点，包括URL参数、表单字段、Cookie和HTTP头（如X-Forwarded-For）。可以使用Nmap扫描开放端口和服务来辅助识别。

漏洞探测阶段，可以采用手工测试和自动化工具相结合的方式：

  • 手工测试：在参数中插入单引号、双引号等特殊字符，观察页面响应。如果出现数据库错误信息，说明可能存在未过滤的注入点；如果返回拦截页面，则说明有WAF防护，需要尝试绕过手法。例如，在登录表单中输入’ OR ‘1’=’1，看是否能绕过认证。
  • 自动化工具：使用SqlMap进行初步扫描，基础命令为sqlmap -u “http://example.com/page.php?id=1“ –batch。对于POST请求，可以使用–data参数指定请求数据。

2、如何利用漏洞及使用工具

确认存在SQL注入漏洞后，利用过程主要包括数据库信息枚举和数据导出：

使用SqlMap进行自动化利用是最常见的方式：

  • 枚举数据库：sqlmap -u “http://example.com/page.php?id=1“ –dbs，获取所有数据库名。
  • 枚举表：sqlmap -u “http://example.com/page.php?id=1“ -D 数据库名 –tables，获取指定数据库中的所有表。
  • 枚举列：sqlmap -u “http://example.com/page.php?id=1“ -D 数据库名 -T 表名 –columns，获取表中的列名。
  • 导出数据：sqlmap -u “http://example.com/page.php?id=1“ -D 数据库名 -T 表名 -C 列名 –dump，导出指定列的数据。

手工利用场景：当自动化工具受限或需要精细化操作时，可以手工构造payload。例如：

  • 联合查询：’ UNION SELECT username, password FROM users –，直接获取用户名密码。
  • 报错注入：利用数据库函数制造错误信息，如MySQL的updatexml()函数。
  • 布尔盲注：通过页面返回的True/False差异逐字符猜解数据。
  • 时间盲注：使用延迟函数（如MySQL的sleep(5)）判断条件是否成立。

辅助工具：Burp Suite用于拦截和修改HTTP请求，通过Intruder模块进行手动测试和payloadfuzz。

3、拿到加密信息如何破解（MD5破解方法）

MD5是一种不可逆的哈希函数，理论上无法直接解密，但可以通过以下方法尝试”破解”（即找到产生相同哈希值的原始数据）：

  • 字典攻击：利用预先构建的包含常见密码的字典（如rockyou.txt），逐个计算MD5值进行比对。这种方法相对较快，成功率取决于字典的质量。
  • 彩虹表攻击：使用预先计算好的彩虹表（明文到哈希的映射表），通过查表方式快速找到原始数据。彩虹表在空间和时间上做了平衡，比字典攻击更快。
  • 在线MD5查询网站：如cmd5.com等，这些网站收集了大量已破解的MD5值，直接查询即可。
  • 暴力破解：尝试所有可能的字符组合，直到找到匹配项。这种方法非常耗时，尤其是对于长密码或复杂密码。

实际场景应对：

  • 如果密码是弱密码（如123456、password等），通常几秒内就能通过字典攻击破解。
  • 如果密码是强密码（长度长、含特殊字符），破解难度极大，可能无法成功。
  • 如果数据库使用了加盐（Salt）：即在密码后添加随机字符串再计算MD5，破解难度会大幅增加，需要同时知道盐值才能构造字典。

4、拿到信息后如何利用

获取到数据库中的信息后，后续利用方向取决于目标：

  • 直接登录后台：如果获取的是管理员表的用户名和密码（明文或已破解），尝试登录管理后台。
  • 横向移动：获取的密码可能在其他系统（邮箱、VPN、OA系统）中被重用，尝试用相同密码登录其他服务。
  • 数据贩卖/勒索：导出用户个人信息、信用卡数据、订单记录等敏感数据。
  • 权限提升：如果数据库账号权限较高（如root或sa），可以尝试通过数据库功能执行系统命令、写入webshell，进而控制服务器。例如，MySQL的into outfile功能可以写入文件。
  • 持久化访问：在系统中创建后门账户，或植入webshell，保持长期访问权限。

5、为什么管理员账号密码登录不上管理界面

这是一个非常常见的问题，即使拿到了正确的用户名和密码，登录失败通常由以下原因导致：

  • 密码哈希问题：

    • 未知的加盐方式：应用程序可能在存储密码时使用了加盐哈希（如md5($salt.$password)），你拿到的只是最终哈希值，即使破解了也可能不是原始密码。
    • 多次哈希：应用可能对密码进行了多次哈希（如md5(md5($password))），你需要了解具体的哈希逻辑才能构造正确的登录请求。
    • 非明文存储：有些系统根本不存储密码，而是存储密码的哈希值，登录时对用户输入进行同样的哈希运算后比对。即使你知道原始密码，也要确保登录请求的提交方式与系统预期一致。

  • 应用层限制：

    • IP绑定：管理员账户可能被限制只能从特定IP登录，你的IP不在白名单中。
    • 双因素认证：后台开启了二次验证，需要输入手机验证码或Google Authenticator码。
    • Session/Cookie验证：登录成功后，系统可能还需要验证额外的Session或Cookie信息。
    • 登录验证码：登录界面有验证码，自动化登录被拦截。

  • 权限不匹配：

    • 数据库账号 ≠ 应用账号：你从数据库users表拿到的账号，可能只是普通用户账号，根本不是管理员账号（尽管表名可能叫”admin”）。真正的管理员账号可能存储在另一个表或另一个数据库中。
    • 账号被禁用：该管理员账号可能已被系统管理员禁用或删除。

  • 绕过登录≠正常登录：如果你是通过SQL注入直接绕过登录验证进入后台（如’ OR ‘1’=’1’ –），这只是在查询层面绕过了验证，并不代表你掌握了真正的管理员凭证。当你试图用”推测”出的密码正常登录时，系统会进行完整验证，自然无法通过。

7、 请描述一次完整的HTTP请求过程

DNS 解析域名得到 IP 地址。

与服务器建立 TCP 连接（三次握手）。

客户端发送 HTTP 请求。

服务器处理请求并返回 HTTP 响应。

浏览器解析渲染页面。

关闭 TCP 连接（四次挥手）或保持长连接。

8、 请描述Cookies和session区别是什么？

Cookie：客户端存储机制，由服务器发送并保存在浏览器，用于会话跟踪，大小有限，安全性较低。

Session：服务器端存储机制，保存用户状态信息，通过 Cookie 中的 session ID 关联，相对安全但占用服务器资源。

9、 请描述GET 和 POST 的区别是什么？

GET：请求数据附在 URL 后，有长度限制，幂等，可缓存，用于获取资源。

POST：请求数据在请求体中，无长度限制，非幂等，一般不缓存，用于提交数据。

安全性上 POST 稍好，但均未加密。

10、请描述HTTPS和HTTP的区别是什么？

HTTP：明文传输，默认端口80，无加密。

HTTPS：在 HTTP 和 TCP 之间加入 SSL/TLS 加密层，默认端口443，提供数据加密、身份认证和完整性保护，需要证书，安全性高但速度稍慢。

11、请描述session 的工作原理？

当用户首次访问服务器时，服务器为其创建唯一的 session ID，并通过 Cookie 发送给客户端保存。后续请求中，客户端携带该 session ID，服务器根据 ID 查找对应的 session 数据（可存储在内存、文件或数据库中），从而识别用户状态。

12、 http长连接和短连接的区别是什么？

短连接：每次 HTTP 请求都建立新的 TCP 连接，完成后关闭，开销大。

长连接（HTTP keep-alive）：一次 TCP 连接可处理多个 HTTP 请求，减少连接建立开销。HTTP/1.1 默认支持长连接，通过 Connection: keep-alive 控制

13、 请描述OSI 的七层模型都有哪些？

物理层、数据链路层、网络层、传输层、会话层、表示层、应用层。

14、 请描述session 的工作原理？什么是TCP粘包/拆包？发生原因？解决方案

Session 工作原理：当用户首次访问服务器时，服务器为其创建唯一的 session ID，并通过 Cookie 发送给客户端保存。后续请求中，客户端携带该 session ID，服务器根据 ID 查找对应的 session 数据（可存储在内存、文件或数据库中），从而识别用户状态。

TCP 粘包/拆包：TCP 是基于字节流的协议，无消息边界，可能导致多个发送的数据合并接收（粘包）或一个数据被拆分接收（拆包）。

发生原因：Nagle 算法、流量控制、缓冲区大小、网络延迟等。

解决方案：应用层添加消息边界，如固定长度、特殊分隔符、消息头携带长度字段等。

15、 URI和URL的区别是什么？

URI（统一资源标识符）：用于标识某一资源的字符串。

URL（统一资源定位符）：是 URI 的子集，不仅标识资源，还提供访问方式（如协议、主机等）。

例如：https://example.com/path 是 URL，也是 URI；urn:isbn:0451450523 是 URI 但不是 URL。

16、 什么是SSL ？https是如何保证数据传输的安全？

SSL（安全套接层）是加密协议，现多指 TLS。HTTPS 通过 SSL/TLS 在传输层加密数据：

使用非对称加密交换会话密钥。

使用数字证书验证服务器身份。

使用对称加密传输实际数据。

从而保证数据的机密性、完整性和身份认证

17、 https是如何保证数据传输的安全（SSL是怎么工作保证安全的）

SSL/TLS 握手过程：

客户端发起请求，包含支持的加密算法等。

服务器返回数字证书（含公钥）。

客户端验证证书合法性，生成预主密钥，用服务器公钥加密发送。

服务器用私钥解密，双方基于预主密钥生成会话密钥。

后续通信使用会话密钥进行对称加密。

18、 TCP对应的应用层协议，UDP对应的应用层协议

TCP 协议：HTTP、HTTPS、FTP、SMTP、POP3、Telnet、SSH 等。

UDP 协议：DNS、DHCP、TFTP、SNMP、RTP、NTP 等

19、 常见的状态码有哪些？

1xx：信息性，如 100 Continue。

2xx：成功，如 200 OK、201 Created。

3xx：重定向，如 301 Moved Permanently、302 Found、304 Not Modified。

4xx：客户端错误，如 400 Bad Request、401 Unauthorized、403 Forbidden、404 Not Found。

5xx：服务器错误，如 500 Internal Server Error、502 Bad Gateway、503 Service Unavailable。

20、什么是SQL注入攻击？

答：攻击者在HTTP请求中注入恶意的SQL代码，服务器使用参数构建数据库SQL命令时，恶意SQL被一起构造，并在数据库中执行。用户登录，输入用户名lianggzone，密码 ‘ or ‘1’=’1 ，如果此时使用参数构造的方式，就会出现select * from user where name = ‘lianggzone’ and password = ‘’ or ‘1’=‘1’不管用户名和密码是什么内容，使查询出来的用户列表不为空。

如何防范SQL注入攻击

使用预编译的PrepareStatement是必须的，但是一般我们会从两个方面同时入手。

Web端

1）有效性检验。

2）限制字符串输入的长度。

服务端

1）不用拼接SQL字符串。

2）使用预编译的PrepareStatement。

3）有效性检验。(为什么服务端还要做有效性检验？第一准则，外部都是不可信的，防止攻击者绕过Web端请求)4）过滤SQL需要的参数中的特殊字符。比如单引号、双引号。

21、 什么是XSS攻击？

跨站点脚本攻击，指攻击者通过篡改网页，嵌入恶意脚本程序，在用户浏览网页时，控制用户浏览器进行恶意操作的一种攻击方式。

如何防范xss攻击？

1）前端，服务端，同时需要字符串输入的长度限制。

2）前端，服务端，同时需要对HTML转义处理。将其中的”<”,”>”等特殊字符进行转义编码。防XSS 的核心是必须对输入的数据做过滤处理。

22、什么是CSRF攻击？

跨站点请求伪造，指攻击者通过跨站请求，以合法的用户的身份进行非法操作。可以这么理解CSRF攻击：攻击者盗用你的身份，以你的名义向第三方网站发送恶意请求。

CRSF能做的事情包括利用你的身份发邮件，发短信，进行交易转账，甚至盗取账号信息。

如何防范CSRF攻击

安全框架，例如Spring Security。oken机制。在HTTP请求中进行token验证，如果请求中没有token或者token内容不正确，则认为CSRF攻击而拒绝该请求。

验证码。通常情况下，验证码能够很好的遏制CSRF攻击，但是很多情况下，出于用户体验考虑，验证码只能作为一种辅助手段，而不是最主要的解决方案。referer识别。在HTTP Header中有一个字段Referer，它记录了HTTP请求的来源地址。如果Referer是其他网站，就有可能是CSRF攻击，则拒绝该请求。但是，服务器并非都能取到Referer。很多用户出于隐私保护的考虑，限制了Referer的发送。

在某些情况下，浏览器也不会发送Referer，例如HTTPS跳转到HTTP。

1）验证请求来源地址；

2）关键操作添加验证码；

3）在请求地址添加 token 并验证。

23、什么是文件上传漏洞

文件上传漏洞指的是用户上传一个可执行的脚本文件，并通过此脚本文件获得了执行服务端命令的能力。许多第三方框架、服务，都曾经被爆出文件上传漏洞，比如很早之前的Struts2，以及富文本编辑器等等，可被攻击者上传恶意代码，有可能服务端就被人黑了。

如何防范文件上传漏洞

文件上传的目录设置为不可执行。

1）判断文件类型。在判断文件类型的时候，可以结合使用MIME Type，后缀检查等方式。因为对于上传文件，不能简单地通过后缀名称来判断文件的类型，因为攻击者可以将可执行文件的后缀名称改为图片或其他后缀类型，诱导用户执行。

2）对上传的文件类型进行白名单校验，只允许上传可靠类型。

3）上传的文件需要进行重新命名，使攻击者无法猜想上传文件的访问路径，将极大地增加攻击成本，同时向shell.php.rar.ara这种文件，因为重命名而无法成功实施攻击。

4）限制上传文件的大小。

5）单独设置文件服务器的域名。

24、DDos 攻击

客户端向服务端发送请求链接数据包，服务端向客户端发送确认数据包，客户端不向服务端发送确认数据包，服务器一直等待来自客户端的确认没有彻底根治的办法，除非不使用TCPDDos

预防：

1）限制同时打开SYN半链接的数目2）缩短SYN半链接的Time out 时间3）关闭不必要的服务

25、防御和检查SQL注入的主要手段有哪些？

  • 参数化查询（预编译语句）：将SQL代码与数据分离，从根本上杜绝注入。
  • 输入验证与过滤：采用白名单校验数据类型、格式和长度，对特殊字符进行转义。
  • 最小权限原则：数据库账户仅授予必要的操作权限，避免使用高权限账户连接应用。
  • 错误信息屏蔽：自定义错误页面，不暴露数据库结构或具体SQL错误。
  • 使用ORM框架：大多数现代ORM内置了防注入机制。
  • Web应用防火墙（WAF）：通过规则匹配拦截恶意SQL语句。
  • 定期漏洞扫描与渗透测试：主动发现潜在注入点。
  • 安全编码规范：避免动态拼接SQL语句。

26、什么是网络安全中的双因素认证？

双因素认证（2FA）是一种身份验证机制，要求用户提供两种不同类别的凭证来确认身份。通常包括以下三类中的两类：

  • 知识因素：用户知道的（如密码、PIN码）。
  • 持有因素：用户拥有的（如手机令牌、硬件Key、智能卡）。
  • 固有因素：用户生物特征（如指纹、面部识别）。
    作用：即使密码泄露，攻击者也因缺少第二因素而无法轻易登录，显著提升账户安全性。

27、能否解释一下XSS cookie盗窃是什么意思？

XSS（跨站脚本）cookie盗窃是指攻击者利用网站存在的反射型或存储型XSS漏洞，注入恶意脚本。当用户浏览器执行该脚本时，脚本会读取document.cookie，并将用户的会话Cookie发送到攻击者控制的服务器。
后果：攻击者可以使用盗取的Cookie冒充受害者身份登录系统（会话劫持），无需密码即可访问受害者的账户。
预防：设置Cookie的HttpOnly属性（脚本不可读）、使用Secure属性（仅HTTPS传输）、输入输出编码防御XSS。

28、什么是网页挂马？网页挂马都有什么类型？

网页挂马是指在正常网页中通过非法手段嵌入恶意代码，当用户访问时，利用浏览器或其插件的漏洞自动下载并执行木马程序。
常见类型：

  • 框架挂马：使用隐藏的<iframe>引入恶意页面。
  • JS脚本挂马：通过JavaScript动态创建恶意元素或重定向。
  • CSS挂马：利用CSS表达式或背景图等方式触发漏洞。
  • 图片挂马：在图片文件中嵌入恶意代码，利用溢出漏洞。
  • Flash/SWF挂马：在Flash文件中编写恶意ActionScript利用漏洞。
  • 伪装挂马：将恶意链接伪装为正常下载链接或按钮。

29、网页木马的防御和清除

防御措施：

  • 及时更新操作系统、浏览器及插件（Flash、Java等）补丁。
  • 使用杀毒软件与防火墙，开启网页实时防护。
  • 禁用不必要的浏览器ActiveX控件和脚本。
  • 对网站进行代码审计，防止被植入恶意代码。
  • 实施内容安全策略（CSP）限制脚本来源。

清除手段：

  • 使用杀毒软件全盘查杀木马。
  • 检查浏览器插件和启动项，移除可疑项目。
  • 清理Hosts文件、恶意注册表项。
  • 对网站服务器：对比备份文件，找出被篡改的页面及新增文件，删除后门代码，修复漏洞。
  • 修改所有可能泄露的FTP、数据库密码。

32、渗透测试的基本流程

前期交互：与客户明确测试范围、规则、时间窗口和授权。

信息收集：主动/被动收集目标域名、IP、子域名、开放端口、服务版本等。

威胁建模与漏洞扫描：利用工具（Nessus、OpenVAS）和手工检测，识别潜在漏洞。

漏洞利用：验证漏洞，获取初步访问权限，尝试提权与横向移动。

后渗透维持：权限维持、数据收集、痕迹清理、权限拓展。

分析报告：整理发现的漏洞、利用路径、风险等级，给出修复建议。

33、什么是入侵检测系统？

入侵检测系统（IDS）是对网络传输进行实时监控，在发现可疑行为或攻击时发出警报的硬件或软件系统。

  • 基于网络（NIDS）：监听网络流量，匹配攻击特征。
  • 基于主机（HIDS）：监测主机日志、文件完整性等。
  • 检测方法：特征匹配（签名）和异常检测。
    IDS 属于被动监听，不阻断流量；不同于IPS（入侵防御系统）会主动拦截。

34、请说明DES算法的基本过程

DES（数据加密标准）为对称分组密码，以64位分组和56位有效密钥。
基本过程：

初始置换（IP）：将64位明文按固定表重新排列。

密钥轮生成：56位密钥经过置换选择1、循环左移、置换选择2，产生16个48位子密钥。

16轮Feistel结构：将IP后的数据分成左右各32位。每轮：

  • 右半部分经扩展置换E变为48位；
  • 与子密钥异或；
  • 经S盒代换（8个S盒，各6入4出）得到32位；
  • P盒置换后与左半部分异或，左右交换。

逆初始置换（IP⁻¹）：最后一轮不交换左右，直接进行IP⁻¹得到密文。
（已不再安全，被AES取代）

35、代理服务器作用

  • 隐藏真实IP：保护客户端匿名性，突破访问限制。
  • 加速与缓存：缓存常用内容，减少带宽和提高响应速度。
  • 访问控制：按策略过滤网页内容，实现上网行为管理。
  • 安全隔离：作为内网和外网的中间层，阻断直接连接，可检测恶意流量。
  • 负载均衡：分发请求到多台后端服务器。

36、简述对称密钥密码体制的原理和特点

原理：加密和解密使用同一个密钥，或可由密钥简单导出。发送方使用密钥将明文变为密文，接收方用同一密钥还原。
特点：

  • 效率高、速度快，适合大数据量加密。
  • 密钥管理困难：N个用户需管理N(N-1)/2个密钥，分发需要安全信道。
  • 无法天然提供不可否认性。
  • 常见算法：AES、DES、3DES。

37、常见加密密钥分配有几种方案，请对比它们的优劣势

通常指密钥分配方式，常见方案：

人工预分配：当面交付或安全物理介质分发。

  • 优势：安全性高，无网络风险。
  • 劣势：难以大规模部署，效率低。

基于密钥分发中心（KDC）：由可信第三方生成并分发临时会话密钥（如Kerberos）。

  • 优势：集中管理，密钥可按需生成，用户只需与KDC共享长期密钥。
  • 劣势：单点故障，KDC成为攻击焦点。

基于公钥证书（PKI）：用非对称加密协商对称密钥（如TLS握手）。

  • 优势：大规模开放系统适用，无需共享预置密钥，可验证身份。
  • 劣势：证书管理复杂，计算开销大。

预共享密钥结合密钥派生：如蓝牙配对。

  • 优势：简单。
  • 劣势：初始秘密脆弱，扩展性差。

38、解释身份认证的基本概念

身份认证是证实一个实体（用户、设备或进程）所声明身份的过程。它通过验证凭据确定“你是谁”。
认证因素包括知识型、所有型、固有型。认证方式可以是单因素、双因素或多因素。其目的在于防止未授权访问，是访问控制的基础。

39、电子邮件存在哪些安全漏洞

  • 垃圾邮件与钓鱼邮件：恶意链接、诱导信息导致信息泄露。
  • 恶意附件：携带病毒、木马或勒索软件。
  • 明文传输：SMTP/POP3/IMAP未启用TLS，邮件内容被嗅探。
  • 发件人伪造：SMTP协议缺乏强制身份验证，可伪造任意发件人（需SPF/DKIM/DMARC防护）。
  • 邮件服务器漏洞：开放中继被利用发送垃圾邮件，服务器软件漏洞被攻击。
  • 账户密码泄露：暴力破解或撞库导致邮箱被完全接管。
  • 机密性缺失：邮件在服务端可能被管理员查看，端到端加密未普及。

40、防火墙应满足的基本条件是什么？

所有进出流量必须经过防火墙：它是网络唯一的咽喉点。

只允许经过安全策略授权的流量通过：严格执行“默认拒绝”或“最小权限”。

自身具备高抗攻击能力：防火墙操作系统应强化加固，免受威胁。

提供有效的审计和告警功能：记录流量和违规事件。

41、列举防火墙的几个基本功能

  • 包过滤：基于IP、端口、协议过滤数据包。
  • 状态检测：维护连接状态表，放行合法会话的返回流量。
  • 应用代理/网关：对应用层协议进行解析过滤（如HTTP、FTP）。
  • 网络地址转换（NAT）：隐藏内部IP，节约公网地址。
  • VPN支持：建立加密隧道。
  • 入侵防御：集成IPS功能，深度检测载荷。

42、静态包过滤和动态包过滤有哪些区别？

  • 静态包过滤：每个数据包独立检查，基于ACL（源/目的IP、端口、协议）匹配，无连接状态概念。效率高但无法关联流量方向，需为返回流量开放大段端口，安全性低。
  • 动态包过滤（状态检测）：维护连接状态表，识别数据包所属会话。允许出站后，自动放行相关的返回包，更安全、灵活，是主流防火墙核心技术。

43、什么是同源策略？

同源策略是浏览器最核心的安全机制，它限制一个源的文档或脚本如何与另一个源的资源交互。“源”由协议、主机（域名）和端口三者共同定义，三者完全相同视为同源。
限制：不同源的文档无法通过JavaScript读取对方的Cookie、DOM、localStorage，Ajax请求也受严格限制（跨域需CORS等机制支持）。这有效防止了恶意网站窃取数据。

44、SYN攻击原理

SYN攻击（SYN Flood）是一种典型的DDoS攻击，利用TCP三次握手的缺陷。
过程：攻击者向服务器发送大量伪造源IP的SYN包。服务器回应SYN-ACK并分配资源进入半连接状态，但因源IP虚假，不会收到最终ACK。半连接队列被迅速占满，导致正常用户无法建立新连接。
防御：SYN Cookie、SYN Cache、增加队列、减少超时时间、防火墙限流等。

45、IIS服务器应该做哪些方面的保护措施？

  • 及时更新：安装微软发布的安全补丁和更新，避免漏洞利用。
  • 最小化安装：仅安装必要的角色和服务模块（如ASP.NET、静态内容等），禁用不必要的ISAPI筛选器。
  • 删除默认站点与内容：移除默认网站、虚拟目录及示例程序。
  • 权限最小化：应用程序池使用低权限内置账户（如ApplicationPoolIdentity），NTFS文件权限严格控制。
  • 配置安全：禁用目录浏览，设置严格请求过滤限制，关闭不必要的HTTP方法（如TRACE），自定义错误页隐藏服务器版本。
  • 使用HTTPS：绑定SSL证书，强制TLS 1.2/1.3。
  • 日志监控：启用详细日志并定期审计，配置WAF和IP限制。
  • 专用应用池与隔离：不同网站使用独立应用池，设置回收与限制策略。
  • 反扫描措施：安装并配置IIS Request Filtering和动态IP限制扩展，抵御暴力破解和扫描。
